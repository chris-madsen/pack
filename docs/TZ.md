## 1. Введение и цель проекта

`pack` — это проект экспериментальной **lossless-системы сжатия**, в которой исходный блок данных представляется не через классический поиск совпадений, словарь или статистическую модель, а через:

- `K` — **магическую константу**, то есть компактное описание производящей функции;
- `V` — **вектор чётности / breadcrumbs**, достаточный для точного обратного восстановления;
- `overhead` — служебные данные слоя и контейнера.

Целевая модель проекта:

1. Для блока `N` сгенерировать компактную производящую функцию.
2. Представить её в сериализуемом виде как `K`.
3. Получить `V`, который задаёт обратимый путь восстановления.
4. Рекурсивно упаковывать поток `(K, V)` следующими слоями.

Проект исходит из принципа:

```text
сжатие = генерация короткого алгоритмического описания + короткий вектор восстановления
```

Ключевая идея: система не должна “угадывать” функцию через большой внешний поиск; она должна **порождать** её по детерминированным правилам на основе анализа блока.

## 2. Словарь терминов

### 2.1. Исходный блок `N`

Фрагмент входного битового потока, который рассматривается как единица кодирования на конкретном слое.

### 2.2. Магическая константа `K`

Компактное кодовое представление производящей функции или композиции функций, достаточное для восстановления алгоритма преобразования блока.

### 2.3. Вектор чётности `V`

Битовое представление ветвлений/шагов/крошек, которое требуется декодеру, чтобы восстановить исходный блок точно и детерминированно.

### 2.4. Метагенератор

Детерминированная система правил, которая из анализа блока строит производящую функцию и её представление `K`.

### 2.5. Производящая функция

Алгоритм, задающий способ перехода между компактным представлением и исходным блоком, либо описывающий обратимую траекторию преобразования.

### 2.6. Слой сжатия

Один проход системы по входному потоку, в результате которого поток блоков преобразуется в поток архивных записей `(K, V, metadata)`.

### 2.7. Самосопряжённый / самообратный оператор

Оператор, для которого повторное применение или применение сопряжённой формы приводит к точному обратному переходу. В проекте это основной строительный блок для обратимых преобразований.

### 2.8. Overhead / служебные данные

Все данные, не относящиеся непосредственно к полезной нагрузке `K` и `V`: заголовки, версии формата, длины, флаги, индексы слоёв, контрольные поля.

## 3. Формальная постановка задачи

Для каждого блока `N` система должна построить представление:

```text
ArchiveBlock = K + V + overhead
```

такое, что:

1. `decode(K, V, overhead) = N`
2. декодирование полностью детерминировано;
3. операции, определяемые `K`, корректны и обратимы;
4. выполняется критерий эффективности:

```text
|N| > |K| + |V| + overhead
```

### 3.1. Что значит “генератор сжатия найден”

Генератор сжатия для блока считается найденным, если:

- по блоку детерминированно построен допустимый `K`;
- по `K` и правилам слоя может быть получен достаточный `V`;
- декодер восстанавливает исходный блок бит-в-бит;
- итоговое представление короче исходного блока по полной честной длине.

### 3.2. Когда слой считается завершённым

Слой считается завершённым, когда все входные блоки преобразованы в слой архивных записей и собран результирующий поток слоя `S_next`.

### 3.3. Когда прекращается рекурсия

Рекурсия слоёв прекращается, если следующий слой не даёт дополнительного выигрыша:

```text
|S_next| >= |S_current|
```

или если достигнут заранее установленный предел по числу слоёв.

## 4. Архитектурные принципы

### 4.1. Генерация по правилам вместо поиска

`pack` не должен опираться на глобальный перебор пространства программ или весов. Вместо этого используется набор детерминированных правил:

```text
анализ блока -> выбор структурного паттерна -> параметризация -> сериализация в K
```

### 4.2. Обратимость как обязательный инвариант

Любое полезное преобразование в проекте должно быть либо:

- непосредственно обратимым;
- либо сопровождаться достаточным `V`, делающим восстановление точным.

Lossy-режимы запрещены.

### 4.3. Композиционность и рекурсивность

Проект строится из композиций простых обратимых подсистем. Результат слоя сам становится входом следующего слоя.

### 4.4. Элементарные битовые операции как базовый алфавит

Базовые примитивы системы должны сводиться к дешёвым битовым операциям и их обратимым композициям.

### 4.5. Детерминированность

Для одинакового входного блока, одинаковой версии формата и одинаковых правил метагенератора должны получаться одинаковые `K` и одинаковые `V`.

### 4.6. Формальная система общей базы `B`

Стороны кодирования и декодирования считаются заранее согласованными относительно общей формальной базы:

```text
B = {
    Σ_op,      # алфавит элементарных операций
    Σ_pat,     # алфавит паттернов / семейств генераторов
    Ax,        # аксиомы системы
    R,         # правила композиции
    Inv,       # правила обратимости
    Param,     # правила вывода параметров K из анализа блока
    Crumb,     # правила построения V
    Ser,       # каноническая сериализация K и V
    Ver        # версия базы
}
```

`B` не передаётся в архиве как полезная нагрузка каждого блока.  
Передаётся только ссылка на версию базы и конкретные результаты компиляции:

```text
base_version + K + V + overhead
```

Допускается многоуровневая схема базы:

```text
B_STD   — стандартная база, встроенная в движок
B_FILE  — файловая база, детерминированно развёрнутая из K_ROOT
B_LAYER — слойная база, детерминированно развёрнутая из summary(S_i)
```

Нормативное требование:

```text
GenerateBase(seed) должен быть полностью детерминированным.
```

Если у Алисы и Боба одинаковые:

```text
base_version, K_ROOT, layer summaries
```

то они обязаны породить одну и ту же базу операторов.

### 4.7. Аксиомы `Ax`

Минимальные аксиомы текущей системы:

1. обе стороны знают одинаковый набор `pattern_id`;
2. каждая операция и каждый паттерн имеют фиксированную семантику;
3. `K` интерпретируется детерминированно;
4. `V` интерпретируется детерминированно;
5. сериализация канонична;
6. допустимы только те композиции, которые обратимы или полностью восстанавливаемы из `V`.

## 5. Модель данных архива

В данной версии ТЗ фиксируется **логическая**, а не окончательная байтовая схема архива.

### 5.1. Формат блока

Логическая запись одного блока:

```text
BlockRecord {
  mode,
  source_bit_length,
  k_bit_length,
  v_bit_length,
  K,
  V,
  block_flags
}
```

Где:

- `mode` — режим блока (`generated`, `terminal`, `raw`, `reserved`);
- `source_bit_length` — длина исходного блока;
- `k_bit_length` — длина `K` в битах;
- `v_bit_length` — длина `V` в битах;
- `block_flags` — флаги особенностей декодирования.

### 5.2. Формат слоя

```text
LayerRecord {
  layer_index,
  block_count,
  block_size_policy,
  block_size_value,
  layer_flags,
  blocks[]
}
```

### 5.3. Формат верхнего контейнера

```text
ArchiveRoot {
  format_version,
  layer_count,
  root_flags,
  root_metadata,
  top_layer_payload
}
```

### 5.4. Служебные поля для декодирования

Минимально обязательны:

- версия формата;
- количество слоёв;
- режим разбиения на блоки;
- длины `K` и `V`;
- режим интерпретации `K`;
- режим интерпретации `V`.

Окончательный wire-format MVP: **length-prefixed binary TLV** с little-endian целыми.

## 6. Спецификация магической константы `K`

### 6.1. Общая идея

`K` — это не одно число в узком смысле и не просто список параметров.  
`K` — это **компактное кодовое описание алгоритма**, которое может включать:

- заголовок верхнего уровня;
- набор сегментов;
- параметры сегментов;
- мини-байткод/IR для отдельных частей;
- константы для обратимых операторов.

### 6.2. Компактная структура `K`

Нормативная форма для MVP:

```text
K = u64
```

Одна 64-битная константа разворачивается общей базой в конкретный алгоритм.
Для spectral-паттерна раскладка фиксирована:

```text
3 × Walsh index[13]
3 × amplitude[5]
3 × sign[1]
peak_count[2]
block_log2[4]
tie_bit[1]
```

Версия базы и тип паттерна задаются контейнером, поэтому не дублируются внутри
`K`.

### 6.3. Семантика верхнего шаблона

Константа трактуется как “main-функция”: фиксированные битовые поля являются
частями алгоритма, как в `docs/magic_example1.md`:

```text
part_1, part_2, part_3
```

и каждая часть отвечает за свой подалгоритм.

### 6.4. Разворачивание алгоритма

- spectral-паттерн читает координаты, относительные амплитуды и знаки пиков;
- operator-паттерн использует всё значение `K` как корень общей базы;
- расписание операций, ARX-параметры, фазовая маска и Feistel-раунды
  детерминированно выводятся локально и не сериализуются.

### 6.5. Верхняя граница на размер `K`

Для MVP принимается ограничение:

```text
|K| = 64 бит
```

Дополнительно сохраняется экономическое ограничение `|K| <= |N| / 4`.

### 6.6. Требования к сериализуемости и декодируемости

`K` обязан:

- однозначно парситься;
- быть versioned;
- содержать всё, что нужно декодеру;
- не зависеть от внешнего скрытого состояния;
- быть пригодным для побитового сравнения и тестирования.

## 7. Семантика вектора чётности `V`

### 7.1. Назначение `V`

`V` — это не XOR-остаток и не произвольная маска.  
`V` — это **вектор ветвлений/чётности/крошек**, который указывает декодеру, как точно пройти обратную траекторию восстановления.

### 7.2. Базовая модель

Для прямого хода система строит последовательность шагов:

```text
s0, s1, s2, ... , st
```

На каждом шаге выделяется один бит или короткая группа бит, характеризующая выбор ветви/режима:

```text
b0, b1, b2, ... , bt
```

Тогда:

```text
V = concat(b0, b1, b2, ... , bt)
```

### 7.3. Нормативная семантика для MVP

В MVP `V` трактуется как **raw parity bitstream** в порядке прямого хода.  
При декодировании чтение `V` производится самим оператором/траекторией, восстановленной из `K`. Внешняя сущность `CRUMB_CFG` как отдельный форматный слой запрещена; точки потребления битов `V` определяются ходом вычислений.

### 7.4. Минимальная достаточность

`V` считается достаточным, если:

- по нему и по `K` можно выбрать единственную допустимую обратную ветвь на каждом шаге;
- итоговая траектория полностью восстанавливает `N`;
- декодеру не требуется никакая внешняя догадка.

### 7.5. Ограничения на рост `V`

Быстрый рост `V` — один из главных рисков проекта.  
Поэтому каждый слой обязан оценивать:

```text
crumb_ratio = |V| / |N|
```

и учитывать его в критерии целесообразности слоя.

## 8. Класс допустимых производящих функций

### 8.1. Общие требования

Производящая функция должна:

- быть алгоритмически выражаемой через `K`;
- иметь строгую семантику;
- быть воспроизводимой декодером;
- обеспечивать точное восстановление при наличии `V`.

### 8.2. Разрешённые примитивы для MVP

В MVP разрешаются следующие базовые операции:

- XOR с константой или маской;
- циклический сдвиг (`rotl` / `rotr`);
- `x ^= x << s`, `x ^= x >> s` в инвертируемых формах;
- сложение/вычитание по модулю `2^w`;
- умножение на нечётную константу по модулю `2^w`;
- перестановка битовых позиций;
- применение Walsh–Hadamard к допустимому векторному представлению;
- фазовое изменение знака/бита по маске.

### 8.3. Условно допустимые примитивы

Операции вроде `OR`, `AND`, очистки и срезов разрешаются **только** если они входят в доказуемо обратимую композицию или если потерянная информация полностью восстанавливается из `V`.

### 8.4. Корректность композиции

Композиция допустима, если:

- каждый шаг сам обратим или компенсируется `V`;
- порядок шагов зафиксирован в `K`;
- типы промежуточных состояний согласованы;
- число раундов явно записано в `K`.

### 8.5. Многошаговый генератор

Многошаговый генератор разворачивается как:

```text
Expand(B_STD, K, block_mode)
```

## 9. Самосопряжённые и самообратные операторы

### 9.1. Роль в системе

Самообратные операторы нужны как безопасный способ строить ядро производящей функции без необходимости хранить отдельный сложный декодер для каждого класса `K`.

### 9.2. Нормативный шаблон оператора

Базовый шаблон для MVP:

```text
U_K = F_K⁻¹ · H · D_K · H · F_K
```

Где:

- `F_K` — обратимое битовое перемешивание, параметризуемое `K`;
- `H` — оператор Walsh–Hadamard;
- `D_K` — фазовая/знаковая маска, параметризуемая `K`.

### 9.3. Требование самообратности

Если выполняется:

```text
H² = I
D_K² = I
F_K обратим
```

то:

```text
U_K² = I
```

Это и есть нормативная цель базового операторного ядра.

### 9.4. Роль Walsh–Hadamard

Walsh–Hadamard используется в двух качествах:

1. как часть обратимого оператора;
2. как анализатор спектральной структуры блока.

### 9.5. Параметризация через `K`

`K` не “содержится внутри кода программы”, а интерпретируется метагенератором так, чтобы породить конкретные `F_K`, `D_K`, траекторный граф чтения `V` и число раундов.

## 10. Метагенератор: правила порождения функции

### 10.1. Входы метагенератора

Метагенератор получает:

- битовый блок `N`;
- статистические профили блока;
- спектральные признаки;
- конфигурацию слоя;
- версию правил.

### 10.2. Базовая схема работы

Нормативный pipeline:

```text
analyze(N)
-> derive feature set P
-> choose main_pattern_id by deterministic rules
-> compile parameters into one u64 K
-> validate invertibility
-> emit K
```

Допускается жадная пошаговая генерация `K`:

```text
K_0 = пусто
на каждом шаге добавляется следующее правило из топологических признаков
правило принимается только если уменьшение |V| больше прироста |K|
```

То есть метагенератор ведёт себя как детерминированный экономический аудитор, а не как поиск по пространству программ.

### 10.3. Генерация вместо перебора

Метагенератор не делает глобальный перебор пространства функций.  
Допускается только:

- выбор между малым конечным числом уже известных паттернов;
- детерминированная параметризация паттерна из признаков блока;
- жёсткий tie-break по фиксированным правилам.

### 10.4. Разрешённые эвристики

Эвристики разрешены **только** в виде детерминированных правил вывода.  
Недопустимы:

- стохастический поиск;
- недетерминированное обучение во время упаковки;
- внешние “угадывающие” оптимизаторы.

### 10.5. Конструкция метагенератора

Метагенератор должен пониматься как компилятор:

```text
MG_B(N) = (K, V)
```

где:

```text
N -> T(N) -> pattern_id -> compact parameter synthesis -> K -> V
```

А именно:

1. **`TopologyExtractor`** строит сигнатуру `T(N)` блока;
2. **`PatternCompiler`** по `T(N)` выбирает семейство генератора из `Σ_pat`;
3. **`ParamSynthesizer`** вычисляет параметры и сериализует их в `K`;
4. **`CrumbBuilder`** строит `V` как вектор ветвлений/выборов;
5. **`LayerPacker`** собирает слой и подготавливает вход для следующего прохода.

Метагенератор не обязан перебирать огромное множество функций; вместо этого он обязан по правилам базы `B` детерминированно порождать конкретный экземпляр кодирования для блока.

Дополнительное нормативное требование:

- если включён режим файловой базы, то сначала вычисляется `K_ROOT`;
- затем строится `B_FILE = GenerateBase(K_ROOT)`;
- затем блоковый `K_i` кодирует путь/параметры внутри `B_FILE`, а не копирует всю базу в архив.


### 10.7. Ограничение вычислительной сложности

Для MVP запрещён сквозной brute-force перебор топологических признаков с асимптотикой `O(N^2)` на полном блоке.

Нормативное требование:

- спектральный анализ и выбор окон должны опираться на ограниченный зонд и процедуры не хуже `O(N log N)` на активном окне;
- метагенератор не должен перебирать произвольные программы или все возможные `K`;
- если анализатору нужен bounded preview, это должно быть явно детерминировано.

### 10.8. Gas Limit декодера

MVP обязан быть безопасным против зацикливания и патологических блоков.

Нормативное требование:

- декодер должен иметь `MAX_DECODER_STEPS`;
- превышение лимита шагов на один блок является критической ошибкой декомпрессии;
- `K` в MVP трактуется как тьюринг-неполный форматный IR с конечным числом шагов.

### 10.6. Диагностический baseline: alphabet breadcrumbs

Для проверки контейнера, битовой упаковки и рекурсивных слоёв допускается диагностический паттерн:

```text
pattern_id = 0x02
ALPHABET_BREADCRUMBS
```

Смысл паттерна:

- `K` хранит алфавит блока и параметры кодирования;
- `V` хранит битовый вектор выбора символов этого алфавита;
- декодер восстанавливает каждый символ по его индексу в алфавите.

Этот паттерн является общим генератором для данных с малой мощностью алфавита и не зависит от конкретного имени файла или заранее известной формулы.

Он **не считается реализацией целевого метагенератора**, потому что:

- не строит производящую функцию из битовой топологии блока;
- не использует `U_K`, Walsh–Hadamard, ARX или Feistel;
- кодирует позицию каждого символа напрямую;
- не обеспечивает требуемого многократного выигрыша на белом шуме.

## 11. Анализ блока перед генерацией

Для MVP фиксируется следующий список признаков:

### 11.1. Битовые профили

- плотность единиц;
- локальная плотность по окнам;
- распределение популяций битов (`popcnt`) по подокнам.

### 11.2. Корреляции и сдвиги

- корреляция блока с `rotl(block, s)`;
- корреляция блока с `rotr(block, s)`;
- выбор локально сильных сдвигов.

### 11.3. Спектральные признаки

- Walsh-спектр;
- индексы наиболее сильных коэффициентов;
- folded-хеш спектра.

### 11.4. Производные и локальные разности

- `block XOR (block >> d)`;
- `block XOR rotl(block, d)`;
- распределение локальных переходов.

### 11.5. Признаки выбора структуры `K`

Метагенератор обязан использовать признаки блока для:

- выбора `main_pattern_id`;
- выбора параметров `F_K`;
- выбора масок для `D_K`;
- выбора конфигурации breadcrumbs.

В минимальном детерминированном варианте:

- сильные Walsh-пики задают фазовые маски `D_K`;
- лучшие shift-correlation задают сдвиги `F_K`;
- `fold64(Walsh(N)) | 1` задаёт нечётный множитель;
- `fold64(derivative_profile(N))` задаёт add-константу;
- `popcnt_profile` должен влиять хотя бы на один сегмент `K`.

## 12. Процедура кодирования одного блока

Нормативная процедура MVP:

1. Принять входной блок `N`.
2. Привести его к рабочему представлению слоя.
3. Посчитать профили блока.
4. Сгенерировать `K`.
5. Построить по `K` производящую функцию / траекторию сжатия.
6. Получить `V`.
7. Оценить полную длину `K + V + overhead`.
8. Сформировать `BlockRecord`.

### 12.1. Правило формирования `V`

В MVP `V` формируется как конкатенация битов ветвления прямого хода.

Нормативный анти-подменный инвариант из `docs/metaquailty.md`:

- для операторных/траекторных паттернов `V` не должен вырождаться в полный поблочный `N XOR G`;
- `V` должен описывать ветвления, фазы, адреса траекторий или иные компактные критические решения;
- если для выбранного паттерна `V` по смыслу превращается в полноразмерный residual исходного блока, такой паттерн считается нерентабельным и обязан быть отвергнут.

### 12.2. Проверка критерия сжатия

Блок считается успешно сжатым на текущем слое, если:

```text
|K| + |V| + overhead < |N|
```

Если используется порог рентабельности слоя `τ`, блок/слой принимается только если:

```text
gain = 1 - (|K| + |V| + overhead) / |N| >= τ
```

Для блочного MVP:

```text
τ = 0
```

При этом неравенство остаётся строгим: равный размер не принимается. Полный
рекурсивный слой также принимается только при уменьшении контейнера со всеми
заголовками.

### 12.3. Терминальный режим

Формат MVP обязан поддерживать режим `terminal/raw` для случаев, когда дальнейшее сжатие слоя нецелесообразно.

Нормативная цель проекта при этом остаётся такой:

- для целевого класса данных первый слой должен уметь строить нетривиальное представление `K + V`;
- верхние слои могут останавливаться, когда выигрыш исчезает.

---


### 12.4. Стоимость raw-блока

Переход в `raw` осуществляется по жёсткому экономическому критерию.

Нормативное требование:

- если `|K| + |V| + overhead >= |N| - raw_overhead`, блок обязан сохраняться как `raw`;
- стоимость заголовка raw-блока должна быть минимальной;
- метагенератору запрещено оставлять блок в “почти невыгодном” сжатом режиме.

## 13. Процедура декодирования одного блока

Нормативная процедура MVP:

1. Прочитать `BlockRecord`.
2. Если `mode = raw`, вернуть payload без преобразований.
3. Иначе распарсить `K`.
4. По `K` восстановить производящую функцию.
5. Прочитать `V`.
6. Выполнить обратный проход с использованием `V`.
7. Восстановить исходный блок `N`.
8. Проверить длину и, если включено, контрольный хеш.

Декодер не должен использовать никакое внешнее состояние, кроме записанного в архиве.

---

## 14. Многоуровневая рекурсивная упаковка

### 14.1. Первый проход

Исходный поток `S0` режется на блоки и превращается в поток записей:

```text
S0 -> Layer0 -> S1
```

### 14.2. Повторная упаковка

Далее:

```text
S1 -> Layer1 -> S2
S2 -> Layer2 -> S3
...
```

### 14.3. Критерий продолжения

Следующий слой создаётся только если:

```text
|S_next| < |S_current|
```

с учётом полного overhead слоя.

### 14.4. Верхний контейнер

Верхний контейнер обязан хранить:

- число слоёв;
- политику блоков по слоям;
- версии правил;
- payload верхнего выигравшего слоя.

---

## 15. Политика выбора размера блока

### 15.1. MVP

Для MVP фиксируется **постоянный размер блока 4096 бит**.

Причины выбора:

- степень двойки;
- удобство для Walsh–Hadamard;
- умеренный overhead относительно полезной нагрузки;
- достаточный объём для выявления спектральной структуры.

### 15.2. Целевая модель

Целевая модель проекта — **переменный или адаптивный размер блока**.

### 15.3. Ограничения выбора блока

Размер блока должен выбираться так, чтобы:

- `K` не съедал выигрыш;
- `V` не разрастался слишком быстро;
- overhead слоя был мал относительно полезной длины;
- анализ блока оставался вычислительно разумным.

### 15.4. Кандидаты на будущую формулу

В будущих версиях допускается выбор блока из множества степеней двойки:

```text
256, 512, 1024, 2048, 4096, 8192 ...
```

по оценочной функции:

```text
score(B) = predicted_K_bits(B) + predicted_V_bits(B) + overhead(B)
```

с выбором блока, минимизирующего `score(B)`.

## 16. MVP-спецификация

В MVP фиксируются следующие решения:

### 16.1. Формат `K`

Одна магическая константа:

```text
u64
```

Смысл полей определяется версией общей базы и режимом блока в контейнере.
Декодер обязан отвергать любую ширину `K`, отличную от восьми байт.

### 16.2. Алфавит операций

- XOR;
- циклические сдвиги;
- xorshift;
- сложение/вычитание mod `2^w`;
- умножение на нечётную константу;
- перестановки;
- Walsh–Hadamard;
- фазовые маски.

### 16.3. Размер блока

```text
BLOCK_BITS_MVP = 4096
```

### 16.4. Режим `V`

```text
V_MODE_MVP = raw parity bitstream
```

### 16.5. Контейнер

```text
CONTAINER_MVP = binary TLV, little-endian
```

### 16.6. Базовый паттерн

Основной паттерн первого прототипа:

```text
pattern_id = 0x01
U_K = F_K⁻¹ · H · D_K · H · F_K
```

Резервный паттерн:

```text
pattern_id = 0x00
raw/terminal passthrough
```

### 16.7. Диагностический паттерн контейнера

Для ранней проверки wire-format допускается паттерн:

```text
pattern_id = 0x02
ALPHABET_BREADCRUMBS
```

где:

- `K` содержит мощность алфавита, сам алфавит и ширину индекса;
- `V` содержит packed bitstream индексов;
- критерий выбора паттерна полностью детерминирован по сигнатуре блока.

Паттерн `0x02` не закрывает требования MVP к метагенератору. MVP считается реализованным только после появления исполнимого паттерна `0x01`, построения `K` из анализа блока и проверки самообратности `U_K`.

## 17. Критерии корректности

Корректная реализация обязана удовлетворять всем пунктам:

1. Полная обратимость.
2. Детерминированность кодирования.
3. Детерминированность декодирования.
4. Независимость от скрытого внешнего состояния.
5. Однозначность парсинга формата.
6. Инвариантность порядка сегментов в `K`.
7. Совпадение исходного и восстановленного блока побитово.

### 17.1. Недопустимые состояния

Недопустимы:

- `K`, который нельзя распарсить;
- `V`, длина которого не согласована с траекторией/числом точек ветвления, определённых `K`;
- оператор без доказуемой обратимости;
- слой, в котором декодер требует незафиксированную эвристику;
- неоднозначная интерпретация сегментов.

## 18. Тестовая стратегия

### 18.1. Юнит-тесты на операции и обратимость

Проверяются:

- обратимость `rotl/rotr`;
- обратимость xorshift-последовательностей;
- обратимость умножения на нечётные константы;
- корректность фазовых масок;
- корректность прямого/обратного Walsh-блока.

### 18.2. Тесты на сериализацию `K`

- parse -> serialize -> parse;
- сравнение семантики до и после сериализации;
- проверка точной ширины 64 бита;
- проверка влияния каждого семантического поля на развёрнутый алгоритм;
- отказ на неканонической или неверной длине.

### 18.3. Тесты на восстановление из `V`

- корректность чтения parity bitstream;
- однозначность выбора ветвей;
- совпадение исходного блока после полного обратного прохода.

### 18.4. Сквозные тесты

- кодирование одного блока;
- декодирование одного блока;
- кодирование/декодирование слоя;
- остановка многослойной рекурсии;
- корректный `raw/terminal` fallback.

### 18.5. Тесты на примерах из документации

В тестовый набор должны быть включены примеры и идеи из:

- `docs/magic_example1.md`
- `docs/compress_example.md`
- `docs/zip.md`

как минимум в виде проверок на:

- корректный парсинг гибридной `K`;
- воспроизводимость смысла сегментов;
- соответствие документированным инвариантам.

## 19. Метрики и критерии успеха

В прототипе измеряются:

- коэффициент сжатия;
- размер `K`;
- размер `V`;
- размер полного overhead;
- глубина слоёв;
- время кодирования;
- время декодирования;
- число блоков, успешно упакованных нетривиальным `K`.

### 19.1. MVP-метрика успеха

MVP считается успешным, если:

1. формат `K` и `V` стабилен;
2. декодер восстанавливает данные побитово;
3. многоуровневый контейнер работает;
4. хотя бы на части целевых данных достигается честное сжатие:

```text
|archive| < |input|
```

## 20. Ограничения, риски и открытые вопросы

### 20.1. Риск разрастания `K`

Если `K` слишком подробно кодирует алгоритм, выигрыш исчезает.

### 20.2. Риск разрастания `V`

Если breadcrumbs почти повторяют исходную информацию, система перестаёт сжимать.

### 20.3. Граница применимости

На практике система должна искать классы данных, для которых алгоритмическое описание действительно короче полного побитового представления.

### 20.4. Адаптивные блоки

Пока не доказано, какой алгоритм выбора размера блока оптимален на каждом слое.

### 20.5. Класс генераторов

Пока не зафиксировано, какой набор `main_pattern_id` будет достаточным и минимальным.

### 20.6. Нормативные open questions

Подлежат дальнейшей формализации:

- точная семантика траекторного потребления `V` для разных паттернов;
- финальный бинарный wire-format;
- правила оценки перспективности следующего слоя;
- диапазон допустимых версий базы и режимов интерпретации `K`;
- критерий переключения на adaptive block sizing.

## Приложение A. Пример смысловой декомпозиции `K`

Документ `docs/magic_example1.md` показывает желаемую модель:

```text
X = main-pattern
X -> part_1 + part_2 + part_3 + ...
```

где:

- верхняя константа задаёт общий шаблон;
- части задают самостоятельные смысловые подалгоритмы;
- декодер умеет восстановить порядок применения частей без внешних подсказок.

Этот пример считается **иллюстративным**, а не нормативным; однако именно такую гибридную идею должно поддерживать MVP-ТЗ.

## Приложение B. Нормативные решения текущей версии

На момент версии `draft v0.1` приняты следующие решения:

- `K` — одна 64-битная магическая константа;
- `V` — вектор чётности, а не XOR-остаток;
- первый операторный паттерн — `U_K = F_K⁻¹ · H · D_K · H · F_K`;
- MVP работает на фиксированном блоке 4096 бит;
- adaptive block sizing выносится в следующую фазу.

## Appendix C. Executable K amendment (implemented 2026-06-13)

- `K` is exactly one `u64`.
- Spectral encode is part of the real candidate path.
- Spectral `K` stores three weighted Walsh peaks and expands them into a
  deterministic inverse-Walsh predictor without PRNG state.
- Spectral `V` is a canonical stream of increasing exception addresses.
- Operator `K` is a root from which `B_STD` expands the reversible schedule,
  phase, and Feistel parameters.
- Operator execution uses the exact binary butterfly kernel
  `U_K = F_K^-1 · H^-1 · D_K · H · F_K`.
- Operator mode is rejected unless it strictly shortens the parity trajectory
  relative to the original words.
- A block is accepted only when `K + V + overhead < N`.

The former zero-valued spectral fallback and codec-generated `>=10x` quality
fixtures are forbidden. Compression quality must be reported on external files.

Current external noise result:

```text
29,722 bytes -> 26,158 bytes
compression factor = 1.1362x
```

The external noise result is still selected by the alphabet baseline. Spectral
is now implemented and testably selected on Walsh-structured inputs, while the
`>=10x` metagenerator quality objective remains open.
