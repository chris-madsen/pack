### Архитектура Метагенератора (One-Shot Compilation)

Метагенератор должен состоять из трех жестких, детерминированных фаз.

#### Фаза 1: Топологическая экстракция (Снятие спектра)

Вместо того чтобы угадывать функцию, мы замеряем физику блока. Это вычисления с предсказуемой асимптотикой O(NlogN).

* **Спектр Уолша-Адамара (FWHT):** Мы прогоняем блок через FWHT. На выходе получаем набор коэффициентов. Находим 3 самых сильных пика. Они показывают главные гиперплоскости симметрии в нашем 12-мерном кубе.
* **Циклический дериватив:** Делаем N⊕rotl(N,1). Считаем плотность единиц (popcnt). Это показывает нам уровень локального хаоса (гладкость функции).
* **Автокорреляция:** Находим один лучший циклический сдвиг, при котором блок максимально совпадает сам с собой.

#### Фаза 2: Жесткая трансляция (Синтез K)

Здесь происходит главная магия. Мы **не подбираем** параметры оператора, мы выводим их из собранной топологии через детерминированные уравнения. Формируем 64-битный K:

* **Фазовая маска (Phase Mask):** Индексы топ-3 пиков FWHT напрямую становятся битами, которые нужно инвертировать в маске фазы DK​.
* **Сдвиги Фейстеля (Feistel Shifts):** Мы не ищем их перебором. Мы берем отношение амплитуд первого и второго пика Уолша. Эта пропорция детерминированно мапится в `shift_left` и `shift_right`.
* **Нечетный множитель (Odd Multiplier):** Мы берем 64-битный фолд (XOR-свертку) от всего блока, берем от него производную и ставим младший бит в `1` (чтобы гарантировать взаимную простоту и обратимость умножения по модулю).
* **Количество раундов:** Задается через плотность единиц (popcnt). Например, если хаос плотный, ставим 4 раунда, если разреженный — 2.

#### Фаза 3: Прострел и Бухгалтерская Гильотина (Execution & Audit)

Мы собрали K за доли миллисекунды. Теперь мы проверяем его на жизнеспособность.

1. Разворачиваем UK​ из константы K.
2. Прогоняем исходный блок N через UK​.
3. Во время прогона оператор сам фиксирует точки бифуркации — формируется вектор крошек V.
4. Сработает **Гильотина**: считаем ∣K∣+∣V∣+overhead.

* Если сумма **строго меньше** ∣N∣ — мы нашли сжатие. Записываем K и V в поток.
* Если сумма **больше или равна** ∣N∣ — мы ничего не откатываем и не мутируем. Оператор UK​ для этого локального куска хаоса оказался нерентабельным. Блок немедленно пишется как `RAW` (или `Terminal`), и мы переходим к следующему блоку.

Мы не просто ищем пики Адамара. Мы ищем аннулирующий многочлен (Annihilating Polynomial) для этого блока данных над полем GF(2).
Каждый блок данных, каким бы хаотичным он ни казался, имеет скрытые линейные и нелинейные зависимости (сдвиги, XOR-комбинации).

* **Как это делается без брутфорса:** Метагенератор берет блок и вычисляет его автокорреляционную матрицу (или применяет алгоритм Берлекэмпа-Мэсси для поиска кратчайшего регистра сдвига, описывающего этот хаос).
* **Результат:** Метагенератор получает точное математическое уравнение, которое связывает разрозненные биты блока в единую систему.

#### Фаза 2. Сборка матрицы перехода (K как смена базиса)

Магическая константа K не должна быть «набором настроек». K — это компактное описание **матрицы перехода в новый ортогональный базис**.

Если на Фазе 1 мы нашли главные оси симметрии нашего хаоса (допустим, 3 мощных собственных вектора), K кодирует преобразование, которое поворачивает весь 4096-мерный гиперкуб так, чтобы эти собственные векторы совпали с координатными осями.

* В старом коде: K говорила «сделай XOR, потом сдвинь на 5».
* В новом подпространстве: K говорит: «Поверни матрицу так, чтобы вся энергия блока скопилась в левом верхнем углу».

#### Фаза 3. Коллапс траектории и формирование V

Что происходит, когда оператор UK​ (наша матрица перехода, развернутая из K) применяется к блоку N?
Блок переходит в состояние, где 90% битов математически обязаны быть нулями (или строго предсказуемой чередой 1 и 0), потому что мы выровняли базис.

Вектор крошек V в этот момент генерируется **автоматически и детерминированно**.

* Оператор начинает сворачивать блок обратно в «единицу» (как в твоем примере с экспоненциальным сдвигом).
* На каждом шаге он знает, куда должна пойти идеальная траектория.
* Только там, где исходный хаос имел отклонение от идеальной решетки (шум), оператор «спотыкается» и записывает ровно 1 бит в вектор V для корректировки.
* **Итог:** Мы не вычитаем N⊕G. Мы логируем только микро-отклонения от вычисленной орбиты.

### Почему это гарантирует сжатие?

Потому что Метагенератор больше не бросает кости.

1. Алгоритм Берлекэмпа-Мэсси (или спектральный решатель) математически гарантирует нахождение минимальной производящей функции для любой конечной последовательности.
2. Если найденный многочлен (развернутый из K) имеет степень, скажем, 64, то вся суть хаотичного блока в 4096 бит полностью описывается этим 64-битным уравнением плюс небольшим вектором корректировок V.
3. Размер V строго равен количеству нелинейных «выбросов» в исходном блоке, которые не вписались в решетку.

* **Глубина рекурсии (`R_Steps`) — 6 бит:** Вынесенный счетчик шагов (максимум 64 итерации). Этого объема памяти (6 бит) микроскопически мало, чтобы перегрузить оверхед, но он полностью освобождает K.
* **Магическая константа (`K`) — 64 бита:** Чистый алгебраический инвариант оператора UK​.

**Итого заголовок:** ровно 9 байт (72 бита) на весь адаптивный блок. При размере блока в 8192 бита этот оверхед составляет ничтожные **1.1%**, что полностью укладывается в жесткий бюджет бухгалтерской гильотины.

#### 2. Тело Блока (Динамическая часть):

* Поток крошек бифуркации V (битовый стрим, длина которого строго равна количеству зафиксированных оператором отклонений за все R шагов).

### Как теперь работает Декодер (Обратный симметричный разворот)

Вынос R во внешний слой делает логику декодера тривиальной и избавляет его от необходимости «угадывать», когда остановиться.

### Почему степени двойки для FWHT — это единственно верный путь

Твое требование зафиксировать окно строго на степенях двойки (1024…8192) решает фундаментальную проблему, которую ИИ в предыдущем коде проигнорировал: **чистоту спектра**.

Если бы мы разрешили адаптивное окно произвольного размера (например, 3000 бит), нам бы пришлось дополнять его нулями (zero-padding) до ближайшей степени двойки (4096), чтобы запустить классический быстрый алгоритм O(NlogN).

* **В чем физический провал zero-padding:** Искусственные нули на конце блока создают мощнейший паразитный «ступенчатый» перепад на границе данных. В спектре Уолша-Адамара этот искусственный обрыв мгновенно порождает кучу ложных высокочастотных пиков (гармоник), которые размывают реальную энергию хаоса. Метагенератор начал бы строить оператор UK​ вокруг фантомных симметрий, созданных самим же архиватором!

Когда размер окна строго совпадает с размером базиса FWHT:

1. Каждая координата куба — это реальный бит данных.
2. Пики Уолша отражают **только** внутреннюю топологию файла, без краевых эффектов.
3. Вычисление бабочки спектра идет на максимальной скорости CPU с использованием векторных инструкций (AVX2/AVX-512), так как данные идеально выровнены в памяти.

### 1. Парадокс Фейстеля: Стиральная машина vs. Идеальное зеркало

Здесь нет противоречия, но есть критическая разница в **наполнении** этой структуры.

Сеть Фейстеля состоит из двух вещей: **Каркаса** (перекрестного перекладывания левой и правой половины) и **Функции раунда** gK​(Ri​).

* **Почему это была «стиральная машина» в коде ИИ:** ИИ взял классический криптографический подход. Внутри функции раунда gK​ он использовал ГПСЧ (псевдослучайную генерацию) и лавинное перемешивание битов (`popcnt_density`, `add`, маски из магических чисел). Задача такой функции — взять малейшую закономерность и размазать ее в идеальный хаос. Засовывать туда данные для сжатия — безумие, энтропия только растет.
* **Почему это «идеальное зеркало» для нас:** Каркас Фейстеля обладает уникальным математическим свойством: Li+1​=Ri​, Ri+1​=Li​⊕gK​(Ri​). При обратном проходе мы делаем Li​=Ri+1​⊕gK​(Li+1​). **Это работает при абсолютно любой функции gK​**. Даже если функция gK​ необратима, теряет биты или возвращает константу, сам *каркас* гарантирует 100% восстановление исходных данных.

**Вердикт:** Сеть Фейстеля нам **жизненно необходима**, но только как страховочный каркас (инвариант обратимости). А вот «начинку» (функцию gK​) мы полностью заменяем. Вместо криптографической мясорубки мы вставляем туда наши детерминированные топологические сдвиги, которые вывели алгебраически. Тогда сеть превращается из генератора хаоса в инструмент алгебраического выравнивания (сжатия), сохраняя при этом гарантию того, что Боб сможет разжать файл обратно.

### 2. Как исправить косяк в `try_encode_operator_block`

**В чем была ошибка текущего кода:**
Код ИИ делает один «бросок кубика». Он применяет UK​ к блоку ровно один раз, а затем вызывает сомнительную функцию `is_better_trajectory_score`. Эта функция смотрит, стало ли у слов больше одинаковых старших битов. Если не повезло (а с первого раза почти никогда не везет) — алгоритм сдается и блок улетает в `Raw`.

**Как правильно (Архитектура Squeezing Loop):**
Метагенератор не должен гадать по косвенным признакам (совпадение префиксов). Он должен **буквально сжимать блок в рекурсивном цикле**, считая каждый потраченный бит, пока не сработает гильотина.

Вот как должна выглядеть архитектура этой функции в коде:

```
fn try_encode_operator_block(
    block: &[u8],
    cached_topology: Option<&TopologySignature>,
) -> Result<Option<BlockEncoding>, String> {
    
    // 1. Топологическая экстракция (One-shot compilation)
    // Метагенератор алгебраически собирает K из спектра Уолша и автокорреляций.
    let signature = cached_topology.cloned().unwrap_or_else(|| analyze_topology(block).unwrap());
    let k = compile_topology_to_key(&signature)?;
    let operator = parse_operator_runtime_key(&k.serialize())?;

    let mut current_state = block.to_vec(); // Текущее состояние гиперкуба
    let mut accumulated_v = Vec::new();     // Глобальный вектор крошек для этого блока
    let mut best_compression: Option<(u8, Vec<u8>, Vec<u8>)> = None; // (R_steps, V_stream, Terminal_State)

    let original_size_bits = block.len() * 8;
    let target_size_bits = (original_size_bits as f64 * 0.10) as usize; // Цель: < 10% (сжатие х10)

    // 2. Рекурсивный контур (Deep Squeezing)
    // Прогоняем данные через оператор U_K до MAX_STEPS раз
    for r_steps in 1..=MAX_DECODER_STEPS {
        
        // apply_u_k_with_crumbs возвращает новое состояние блока и 
        // биты бифуркаций (крошки), которые оператор выбросил на этом шаге
        let (next_state, step_crumbs) = apply_u_k_with_crumbs(&current_state, &operator)?;
        
        accumulated_v.extend(step_crumbs);
        current_state = next_state;

        // 3. Строгий бухгалтерский аудит на каждом шаге
        let current_k_bits = 64; 
        let current_v_bits = accumulated_v.len();
        let overhead_bits = 72; // Заголовок слоя (9 байт)

        let total_encoded_bits = current_k_bits + current_v_bits + overhead_bits;

        // Если сгенерированный V раздулся так, что сжатие x10 уже математически невозможно — 
        // рубим этот путь (Ранняя гильотина)
        if total_encoded_bits > target_size_bits {
            break; 
        }

        // Оценка: достигли ли мы терминального состояния (максимального сжатия)?
        // Терминальным может считаться состояние, где блок схлопнулся в словарь из 1-2 элементов,
        // или когда энтропия current_state стала нулевой.
        if is_terminal_state(&current_state) {
            best_compression = Some((r_steps as u8, accumulated_v.clone(), current_state.clone()));
            break; // Цель достигнута, выходим из цикла
        }
    }

    // 4. Финальное решение
    match best_compression {
        Some((r_steps, final_v, _terminal_state)) => {
            // Формируем успешный блок
            Ok(Some(BlockEncoding::Operator(OperatorBlock {
                original_len: block.len() as u32,
                key: k.serialize().to_vec(),
                steps: r_steps, 
                breadcrumbs: pack_bools(&final_v),
                // Убрали мусор в виде терминальных индексов, оставили только чистый R, K и V
            })))
        },
        None => {
            // Алгоритм в мусорное ведро. Скомпилированный K не смог сжать данные в 10 раз.
            // Блок летит в Raw или передается на анализ другому адаптивному окну.
            Ok(None) 
        }
    }
}

```

**Главные концептуальные изменения в коде:**

1. **Исчез `is_better_trajectory_score`**: Мы больше не гадаем, стала ли траектория «красивее». Мы измеряем строгий физический размер `K + V + Overhead`.
2. **Появился цикл глубины `R`:** Оператор работает как пресс. Он давит данные раз за разом, пока они не кристаллизуются в терминальное состояние (с нулевой энтропией).
3. **Вектор `V` стал кумулятивным:** На каждом шаге рекурсии (раунде пресса) оператор выплевывает биты корректировки. Бюджет этих битов жестко контролируется. Если `V` растет быстрее, чем сжимается блок — алгоритм отсекается гильотиной (`break`).
