## 1. Базовый самосопряжённый оператор метагенератора

Я бы взял такой оператор:

```text
U_K = F_K⁻¹ · H · D_K · H · F_K
```

Где:

```text
H   = Walsh-Hadamard operator
D_K = phase/sign mask, заданная магической константой K
F_K = обратимое битовое перемешивание, заданное той же K
```

Почему это хороший вариант:

```text
H = H⁻¹
D_K = D_K⁻¹
F_K обратим
```

Значит:

```text
U_K² = I
```

То есть оператор самосопряжённый/самообратный. Один раз применили — получили преобразованное состояние. Второй раз применили — вернулись обратно.

## 2. Как выглядит `H`

Для блока длиной `N = 2^n`:

```text
H[i,j] = (-1) ^ parity(i & j)
```

Parity без отдельной функции:

```text
v = i & j

v ^= v >> 32
v ^= v >> 16
v ^= v >> 8
v ^= v >> 4

bit = (0x6996 >> (v & 0xF)) & 1
```

Это ровно та же логика, которую мы уже использовали для масок простых.

---

## 3. Как выглядит `D_K`

`D_K` — это диагональный оператор знаков:

```text
D_K[j] = (-1) ^ phi_K(j)
```

`phi_K(j)` вычисляется битовой магией из индекса `j` и магической константы `K`.

Пример:

```text
z = j ^ seed

z ^= z << s0
z ^= z >> s1
z *= mul
z += add
z ^= z >> s2

phi = parity(z & mask)
```

Все параметры берутся из `K`:

```text
seed
s0
s1
s2
mul
add
mask
```

`mul` должен быть нечётным, чтобы умножение было обратимым по модулю `2^64`.

---

## 4. Как выглядит `F_K`

`F_K` — это обратимое перемешивание индекса/состояния.

Пример одного слоя:

```text
x ^= x << a
x ^= x >> b
x *= odd_mul
x += add_const
x = rotl(x, r)
```

Операции:

```text
XOR
shift
rotate
+
*
```

Условия:

```text
odd_mul нечётный
rotate обратим
xorshift обратим через обратную последовательность
addition обратима через subtraction
```

Поэтому `F_K⁻¹` существует.

## 5. Полный шаг метагенератора

Пусть `X_t` — текущий блок данных.

Тогда шаг:

```text
X_next = U_K(X_t) XOR B_K(V_t)
```

Где:

```text
K   = магическая константа производящей функции
V_t = хлебная крошка / коррекция на шаге t
B_K = способ разложить крошку в нужный битовый слой
```

Обратный ход:

```text
X_t = U_K(X_next XOR B_K(V_t))
```

Поскольку:

```text
U_K = U_K⁻¹
```

декодер работает той же логикой.

## 6. Магическая константа `K`

Я бы делал не 32-битную, а сразу 128-битную или 256-битную константу. 32 бита слишком быстро упрётся в пространство функций.

Пример 128-битной схемы:

```text
K0: [ block_log ][ rounds ][ op_profile ][ mode ]
K1: [ shifts: s0,s1,s2,s3 ][ rotates: r0,r1 ]
K2: [ masks / spectral selector ]
K3: [ odd_mul ][ add_const ]
```

Универсальные константы ядра:

```text
K_H64   = 0x699606
K_POP64 = 0x55330F01
```

`K_FILE` — уже файл-специфичная магическая константа, которую генерирует анализатор файла.

## 7. Как получать `K` из файла

Вот тут Уолш-Адамар нужен не декоративно, а как анализатор структуры.

Берём блок файла `B`.

Считаем Walsh-спектр:

```text
S = H(B)
```

Дальше из спектра вытаскиваем параметры для `K`.

### 7.1. Сдвиги

Проверяем корреляции:

```text
C_s = similarity(B, rotl(B, s))
```

Где `s` пробегает, например:

```text
1..63
```

Лучшие пики дают:

```text
s0, s1, s2
```

Их кладём в `K`.

### 7.2. XOR / OR / AND-маски

Берём спектральные пики:

```text
top_bits = strongest Walsh coefficients
```

Из них строим:

```text
mask0
mask1
mask2
```

Эти маски потом участвуют в `phi_K`.

### 7.3. Умножитель

Берём spectral hash:

```text
h = fold64(S)
mul = h | 1
```

`| 1` делает множитель нечётным.

### 7.4. Сложение

Берём другой folded-профиль:

```text
add = fold64(S shifted / mixed)
```

Так файл сам диктует:

```text
какие сдвиги
какие маски
какой множитель
какую фазовую функцию
```

## 8. Главная форма оператора

Финальная формула:

```text
U_K = F_K⁻¹ · H · D_K · H · F_K
```

А производящая рекурсия:

```text
X_{t+1} = U_K(X_t) XOR B_K(V_t)
```

Архив:

```text
ARCHIVE = K_FILE + V
```

Где:

```text
K_FILE задаёт производящую функцию
V корректирует остаток
```

## 9. Как это применять к сжатию

Кодирование:

```text
1. Разбить файл на блоки.
2. Для блока посчитать Walsh-спектр.
3. Из спектра собрать K_FILE.
4. Запустить U_K как предиктор/генератор.
5. Остаток записать в V.
6. Если V плотный — поменять K или увеличить число раундов.
7. Если V структурный — прогнать тем же метагенератором рекурсивно.
```

Декодирование:

```text
1. Прочитать K_FILE.
2. Прочитать V.
3. Запустить ту же рекурсию.
4. Применить U_K и XOR-коррекции.
5. Получить исходный блок.
```

## 10. Где здесь popcnt

`popcnt` хорошо использовать внутри `phi_K`, но аккуратно.

Например:

```text
z = j ^ seed
z *= odd_mul
z ^= z >> s

pc = popcnt(z & mask)

phi = (pc ^ (pc >> 1) ^ (z >> r)) & 1
```

Если нужен полностью элементарный вариант без аппаратного `popcnt`, берём SWAR:

```text
BYTE = 0x0101010101010101

P55 = 0x55 * BYTE
P33 = 0x33 * BYTE
P0F = 0x0F * BYTE
P01 = 0x01 * BYTE

z = z - ((z >> 1) & P55)
z = (z & P33) + ((z >> 2) & P33)
z = (z + (z >> 4)) & P0F
pc = (z * P01) >> 56
```

Так `popcnt` тоже превращается в битовую магию.

## 11. Первый рабочий вариант метаоператора

Я бы начал с такого ядра:

```text
K_META_CORE = {
    H:      0x699606,
    ops:    shifts + xor + rotate + add + odd_mul,
    phase:  parity/popcnt mask,
    form:   U_K = F_K⁻¹ · H · D_K · H · F_K
}
```

Минимальная функция:

```text
F_K(x):
    x ^= x << s0
    x ^= x >> s1
    x *= odd_mul
    x += add
    x = rotl(x, r)
    return x
```

Фаза:

```text
phi_K(j):
    z = j ^ seed
    z ^= z << s2
    z *= odd_mul2
    z ^= z >> s3
    return parity(z & mask)
```

Оператор:

```text
U_K(X):
    Y = F_K(X)
    Y = H(Y)
    Y = phase_flip(Y, phi_K)
    Y = H(Y)
    X = F_K⁻¹(Y)
    return X
```

## 12. Куда развивать дальше

Первый уровень — блоки 64 бита.  
Проверяем математику и обратимость.

Второй уровень — блоки 4096 бит.  
Там Walsh-Hadamard уже реально показывает структуру файла.

Третий уровень — adaptive `K`.  
Один блок получает один `K`, группа похожих блоков получает общий `K`.

Четвёртый уровень — recursive residual packing.  
Остаток `V` снова проходит через такой же анализатор.

Пятый уровень — поиск класса функций.  
Метагенератор хранит не один `K`, а маленькую цепочку:

```text
K0, K1, K2
```

и оператор становится палиндромным:

```text
U = U_K0 · U_K1 · U_K2 · U_K1 · U_K0
```

Так сохраняется самосопряжённость всей композиции.

Если условие выполняется — блок сжат.  
Если нет — нужен другой `K`, другой размер блока или fallback.

# chatgpt response

Бро, тут надо поправить архитектуру по уровням. Магическая константа **не должна быть внутри оператора как “часть формулы”**. Правильнее так:

```text
метагенератор = универсальная машина / интерпретатор
K = магическая константа, то есть код конкретной производящей функции
U_K = самосопряжённый оператор, полученный из шаблона U и параметра K
V = хлебные крошки / остаточная коррекция
```

То есть оператор не “содержит” K. Он **параметризуется** через K.

В вашем диалоге с Gemini как раз зафиксирована эта идея: архив — это пара `M + V`, где `M` задаёт закон/функцию, а `V` корректирует траекторию; также там появился Уолш-Адамар как спектральный анализатор файла. 

## 1. Размер магической константы

Фиксировать `K = 32 bit` или `128 bit` заранее не надо.

Правильное условие такое:

```text
|K| + |V| + служебные поля < |N|
```

где `N` — исходный блок белого шума.

Ты говоришь про разницу:

```text
|N| - |V|
```

Но если `K` хранится в архиве, честная разница:

```text
|N| - (|K| + |V| + overhead)
```

Иначе можно “сжать” любой блок так:

```text
K = сам блок N
V = пусто
```

Формально `V` маленький, но сжатия нет.

Поэтому для блока размером, например, `4096 бит` можно разрешить:

```text
K до 256–512 бит
V до 512–1024 бит
```

Если итог меньше `4096 бит`, блок сжат.

---

## 2. Где стоит самосопряжённый оператор

Нужно три уровня.

### Уровень A — универсальный метагенератор

Это фиксированный код программы. Он один для всех файлов.

```text
META_ENGINE
```

Он умеет читать `K` и строить по нему конкретную функцию.

### Уровень B — магическая константа

```text
K
```

Это описание конкретной производящей функции для блока.

Внутри `K` могут быть:

```text
алфавит операций
сдвиги
маски
умножители
сложения
число раундов
режим Уолша-Адамара
режим крошек
```

### Уровень C — оператор

Из шаблона оператора и `K` получается конкретный оператор:

```text
U_K
```

Вот правильная запись:

```text
U_K = F_K⁻¹ · H · D_K · H · F_K
```

Здесь:

```text
F_K  — обратимое битовое перемешивание, заданное K
H    — Walsh-Hadamard
D_K  — фазовая / знаковая маска, заданная K
```

Так как:

```text
H² = I
D_K² = I
```

то:

```text
U_K² = I
```

То есть `U_K` самосопряжённый.

## 3. Как это выглядит в кодовой логике

`K` задаёт `F_K`.

Пример `F_K`:

```text
x ^= x << s0
x ^= x >> s1
x *= odd_mul
x += add
x = rotl(x, r)
```

Все параметры берутся из `K`.

`D_K` задаёт фазу:

```text
z = x ^ seed
z ^= z << s2
z *= odd_mul2
z ^= z >> s3

phi = parity(z & mask)
```

Parity через `0x6996`:

```text
v = z & mask

v ^= v >> 32
v ^= v >> 16
v ^= v >> 8
v ^= v >> 4

phi = (0x6996 >> (v & 0xF)) & 1
```

Сам оператор:

```text
U_K(x) = F_K⁻¹( H( D_K( H( F_K(x) ) ) ) )
```

Это уже нормальный self-inverse cell для метагенератора.

## 4. Рекурсивная схема по слоям

Твоя идея правильная.

Первый слой:

```text
исходный файл
→ блоки N₀, N₁, N₂...
→ для каждого блока получаем Kᵢ + Vᵢ
→ получаем поток S₁ = K₀,V₀,K₁,V₁,K₂,V₂...
```

Второй слой:

```text
S₁
→ снова режем на блоки
→ снова получаем K + V
→ получаем S₂
```

И так несколько раз:

```text
S₀ → S₁ → S₂ → ... → S_L
```

Финальная служебная информация хранится в верхнем контейнере:

```text
K_ROOT
```

Туда кладём:

```text
количество слоёв L
размеры блоков по слоям
алфавит операторов
формат K
формат V
размер последнего слоя
режим восстановления
```

Тогда итоговый архив:

```text
ARCHIVE = K_ROOT + S_L
```

Если получится сильный вариант, можно сделать ещё один метасамосопряжённый оператор:

```text
U_ROOT
```

который работает уже не с одним блоком, а со всей слоистой структурой:

```text
U_ROOT(S_L) → S₀
```

Но первый практический шаг — сделать обычный рекурсивный контейнер. Метасамосопряжённость верхнего уровня можно добавить после того, как заработает блочный `U_K`.

## 5. Про “группа похожих блоков получает общий K”

Ты прав: для белого шума это не основной путь.

Общий `K` для группы нужен только как оптимизация, если несколько блоков случайно хорошо описываются одной функцией. Это не дедупликация и не смысл алгоритма.

Базовая схема должна быть такая:

```text
один блок → свой K → свой V
```

Потом уже можно проверять:

```text
несколько блоков → один K + несколько V
```

Если это даёт выигрыш, используем. Если нет — каждый блок получает индивидуальную функцию.

## 6. Самая важная граница: белый шум

Здесь надо сказать жёстко.

Если под белым шумом понимать **равномерно случайную N-битную строку**, то lossless-сжатие всех таких строк невозможно.

Причина простая:

Всего N-битных блоков:

```text
2^N
```

Если архив имеет размер максимум `A` бит, разных архивов:

```text
2^A
```

Чтобы закодировать все N-битные блоки без коллизий, нужно:

```text
2^A ≥ 2^N
```

значит:

```text
A ≥ N
```

А у нас:

```text
A = |K| + |V| + overhead
```

Поэтому для всех возможных блоков одновременно условие:

```text
|K| + |V| + overhead < |N|
```

выполнить нельзя.

Это не отменяет проект. Это задаёт правильную цель:

```text
для конкретного блока искать короткую производящую функцию K и короткий V
```

То есть мы ищем не универсальный архиватор всех случайных строк, а метагенератор, который находит короткое описание конкретного экземпляра данных.

Если блок реально равномерный случайный, в среднем короткого описания у него нет. Если блок выглядит как белый шум, но имеет скрытый генератор, короткое описание может быть найдено.

## 7. Почему примеры из Markdown не доказывают сжатие всего белого шума

Примеры типа:

```text
маленький вектор → огромное число
```

показывают, что существуют числа с короткой производящей траекторией.

Это сильная идея для генератора.

Но для всех N-битных блоков сразу нужна мощность покрытия:

```text
число возможных пар (K,V) ≥ число возможных блоков
```

Если `K+V` короче `N`, пар меньше, чем блоков. Значит часть блоков обязательно не представима коротко.

Практический смысл проекта остаётся:

```text
найти максимально широкий класс блоков,
которые имеют короткое описание через K + V
```

И сделать поиск `K` достаточно быстрым.

## 8. Как файл должен сам рождать K

Вот тут Уолш-Адамар нужен как анализатор.

Для блока `N` считаем несколько профилей:

```text
Walsh spectrum
битовые производные
корреляции со сдвигами
popcnt-профиль
локальные XOR-разности
```

Из них собираем `K`.

### 8.1. Сдвиги

Ищем пики корреляции:

```text
corr_s = match(N, rotl(N, s))
```

Лучшие `s` попадают в `K`:

```text
s0, s1, s2
```

### 8.2. Маски

Берём сильные Walsh-коэффициенты и превращаем их в маски:

```text
mask0, mask1, mask2
```

### 8.3. Умножители

Берём folded hash спектра:

```text
h = fold64(Walsh(N))
odd_mul = h | 1
```

### 8.4. Сложение

Другой folded-профиль:

```text
add = fold64(derivative_profile(N))
```

### 8.5. Число раундов

Если один раунд даёт плотный `V`, добавляем раунд:

```text
rounds = 2, 3, 4...
```

Именно `rounds` тоже хранится в `K`.

## 11. Как сделать оператор верхнего уровня

Когда несколько слоёв уже получены:

```text
S₀ → S₁ → S₂ → ... → S_L
```

можно построить root-оператор:

```text
U_ROOT = U_{K0} · U_{K1} · ... · U_{KL}
```

Для самосопряжённости делаем палиндром:

```text
U_META =
U_{K0} · U_{K1} · ... · U_{KL} · U_{KL} · ... · U_{K1} · U_{K0}
```

Но такой палиндром может схлопнуться слишком близко к identity, если клетки одинаково применяются к одному регистру.

Лучше делать conjugation:

```text
U_META = F_ROOT⁻¹ · H · D_ROOT · H · F_ROOT
```

где `F_ROOT` строится уже из всей цепочки слоёв.

Тогда верхний оператор остаётся самосопряжённым и описывает весь архив как одну производящую структуру.

Да, это обязательное требование.



```text

если отображение внутри блока инъективно, оно автоматически биективно
```


Именно поэтому база должна быть не пассивным словарём, а **алгеброй производящих функций**.

## 1. Оператор Адамара уже подходит

Новый принципиально другой оператор пока не нужен. Берём тот же класс:

```text
U_K = F_K⁻¹ · H · D_K · H · F_K
```

Где:

```text
H   = Walsh-Hadamard
D_K = отражение / фазовая маска, заданная K
F_K = обратимое перемешивание, заданное K
```

Свойство:

```text
U_K² = I
```

То есть `U_K` самосопряжённый. Это хорошая клетка для метагенератора.

Что меняется по сравнению с простыми числами: раньше `K` задавал конкретную структуру маски, теперь `K` задаёт **производящую функцию для блока данных**.

## 3. Требование биективности

Каждый блок должен кодироваться так:

```text
Encode_B(N) = (K, V)
Decode_B(K, V) = N
```

Требование:

```text
Decode_B(Encode_B(N)) = N
```

Каноническая форма:

```text
N1 ≠ N2  ⇒  Encode_B(N1) ≠ Encode_B(N2)
```

Чтобы это держалось, все основные клетки должны быть обратимыми.

Разрешённые безопасные кирпичи:

```text
x ^= x << s
x ^= x >> s
x = rotl(x, r)
x += c                 mod 2^n
x -= c                 mod 2^n
x *= odd_constant      mod 2^n
```

Умножение только на нечётную константу, потому что оно обратимо по модулю `2^n`.

`popcnt`, `ctz`, `clz` сами по себе необратимы. Их можно использовать внутри Feistel-слоя или внутри фазовой функции `D_K`, где они не уничтожают состояние.

## 4. Как безопасно использовать popcnt и прочую битовую магию

Для этого лучше взять Feistel-слой.

Делим блок на две половины:

```text
L, R
```

Раунд:

```text
L', R' = R, L XOR g_K(R)
```

Функция `g_K` может быть любой:

```text
shifts
xor
or
and
+
*
popcnt
ctz
clz
```

Даже если `g_K` необратима, Feistel-слой всё равно обратим.

Обратный ход:

```text
R = L'
L = R' XOR g_K(L')
```

Поэтому `F_K` можно строить как цепочку Feistel/ARX-слоёв. Это даёт богатство функций и сохраняет биективность.

## 5. Как генерировать производящие функции

Производящая функция — это `F_K` плюс отражение `D_K`.

`K` должен быть переменной длины. Внутри него:

```text
block_log
число раундов
тип клеток
параметры сдвигов
параметры rotate
маски
нечётные множители
add-константы
режим D_K
режим хлебных крошек
```

Пример `F_K`:

```text
F_K(x):
    split x into L, R

    repeat rounds:
        z = R
        z ^= z << s0
        z ^= z >> s1
        z *= odd_mul
        z += add
        z ^= popcnt(z & mask)
        L, R = R, L XOR z

    return join(L, R)
```

Пример `D_K`:

```text
D_K(j):
    z = j ^ seed
    z ^= z << s2
    z *= odd_mul2
    z ^= z >> s3

    phi = parity(z & mask)

    if phi == 1:
        flip sign / flip selected layer
```

Parity через нашу битовую магию:

```text
v = z & mask

v ^= v >> 32
v ^= v >> 16
v ^= v >> 8
v ^= v >> 4

phi = (0x6996 >> (v & 0xF)) & 1
```

## 6. Как из блока получить `K`

Метагенератор должен не перебирать слепо, а строить `K` из самого блока.

Для блока `N` считаем профили:

```text
Walsh-Hadamard spectrum
битовые производные
shift-correlation
popcnt-profile
локальные XOR-разности
```

Из них собираем кандидаты `K`.

### 6.1. Сдвиги

Ищем сильные корреляции:

```text
corr(s) = similarity(N, rotl(N, s))
```

Лучшие `s` идут в поля:

```text
s0, s1, s2
```

### 6.2. Маски

Берём сильные Уолш-Адамар коэффициенты и превращаем их в маски:

```text
mask0, mask1, mask2
```

### 6.3. Множители

Сворачиваем спектр:

```text
h = fold64(Walsh(N))
odd_mul = h | 1
```

### 6.4. Сложение

Сворачиваем производные:

```text
add = fold64(derivative_profile(N))
```

### 6.5. Раунды

Если один раунд даёт длинный `V`, пробуем больше раундов:

```text
rounds = 1, 2, 3, 4...
```

Но раунды увеличивают размер `K` и время, поэтому они принимаются только по gain-правилу.

## 7. Как выглядит кодирование блока

Есть два варианта. Первый проще для прототипа.

### Вариант A: предиктор + остаток

Генератор по `K` создаёт предсказанный блок:

```text
G = Generate(B, K)
```

Остаток:

```text
V = N XOR G
```

Архив:

```text
K + V
```

Декодирование:

```text
N = Generate(B, K) XOR V
```

Это всегда биективно, потому что `V` точно восстанавливает разницу.

### Вариант B: траектория + хлебные крошки

Блок проходит через обратимый/сжимающий путь:

```text
N → Y + V
```

Где `Y` — конечное состояние, `V` — ветки пути.

Архив:

```text
K + Y + V
```

Декодирование:

```text
N = inverse_trajectory(K, Y, V)
```

Этот вариант ближе к Коллатцу и хлебным крошкам. Его стоит делать вторым этапом.

## 8. Что делать с базой

Лучший практический вариант — три уровня базы.

### Уровень 1. Стандартная база `B_STD`

Она встроена в программу и заранее известна Алисе и Бобу.

Содержит:

```text
ARX-клетки
Feistel-клетки
Walsh-Hadamard reflectors
reversible cellular automata
битовые перестановки
несколько prime/E8/решёточных классов как плагины позже
```

Эту базу не храним в архиве. В архиве храним только адреса функций внутри неё.

### Уровень 2. Файловая база `B_FILE`

Она генерируется один раз на файл:

```text
K_ROOT → B_FILE
```

`K_ROOT` хранится в верхнем контейнере.

Файловая база нужна, если много блоков выигрывают от одного набора операторов.

Например:

```text
K_ROOT = spectrum_summary(file)
B_FILE = BaseGen(K_ROOT)
```

Потом каждый блок хранит короткий `K_i`, который ссылается уже на `B_FILE`.

### Уровень 3. Слойная база `B_LAYER`

После первого прохода получается поток:

```text
S1 = K0,V0,K1,V1,...
```

Для следующего слоя можно сгенерировать новую базу:

```text
B_LAYER_1 = BaseGen(summary(S1))
```

И так дальше.

## 9. Генерить базу заранее или для каждого файла

Решение такое:

```text
B_STD — заранее
B_FILE — по необходимости на файл
B_LAYER — по необходимости на слой
```

Стартуем с `B_STD`. Это даст стабильный декодер.

Потом добавляем `B_FILE`, если:

```text
saving_from_shorter_K_i > |K_ROOT| + overhead
```

То есть файловая база хранится только когда она окупается.

Для белого шума это особенно важно: если база на файл реально уменьшает `K_i + V_i`, используем её. Если блоки лучше кодируются индивидуально, работаем через `B_STD`.

Тогда база задаётся маленьким корнем:

Дальше:

`GenerateBase` может сам использовать метагенератор:

```text
B[3] = FeistelMix(B[2])
```

Так база становится не таблицей, а порождаемым семейством операторов.

## 11. Аттракторы, фракталы, Колмогоров

Аттракторы и фракталы могут быть полезны как классы базы, но только в обратимой форме.

Практическая версия:

```text
contractive map + branch breadcrumbs
```

Фрактальная система даёт много самоподобных веток. Хлебные крошки становятся адресами ветвей.

Колмогоровская сложность здесь формулируется относительно базы:

```text
K_B(N) = длина кратчайшего описания N при известной базе B
```

Наша практическая задача:

```text
найти короткое описание N относительно B
```

То есть:

```text
K_B(N) ≈ |K| + |V| + overhead
```

Писать надо так.

Фиксированный код.

```text
B_STD = GenerateStandardBase(version)
```

### 12.3. Block encoder

```text
encode_block(N, B):

    profiles = analyze(N)
        Walsh(N)
        derivatives(N)
        shift_corr(N)
        popcnt_profile(N)

    candidates = synthesize_K(profiles, B)

    for K in candidates:
        U_K = build_operator(K, B)
        V = compute_crumbs_or_residual(N, U_K, B)
        cost = |K| + |V| + overhead

    choose canonical minimum cost

    return K, V
```

### 12.4. Block decoder

```text
decode_block(K, V, B):

    U_K = build_operator(K, B)

    N = reconstruct(K, V, U_K, B)

    return N
```

### 12.5. Layer encoder

```text
S = input_file

for layer in 0..MAX:
    S_next = encode_all_blocks(S)

    gain = 1 - |S_next| / |S|

    if gain < threshold:
        stop

    S = S_next
```

### 12.6. Final archive

```text
ARCHIVE =
    K_ROOT
    layer_map
    final_stream
```

Коротко: оператор Адамара уже есть, менять основу не надо. Метагенератор надо писать как универсальную машину, которая по `K` строит обратимый `U_K`.
