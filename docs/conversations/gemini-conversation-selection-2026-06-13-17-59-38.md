#### 1. Новая формализация: K, V, R

* **K (Magic Constant → ProgramCode):**

* Это **переменная по длине последовательность опкодов** (bytecode).
* Размер не фиксирован. K заканчивается либо по дескриптору длины в заголовке, либо по опкоду `HALT` / `NOP`.
* K — это абстрактное дерево команд (AST), которое декодер интерпретирует для восстановления оператора UK​.
* **V (Branch-Log):**

* Поток корректирующих битов (parity-log).
* Читается декодером только в моменты, когда `ProgramCode` доходит до опкода `BranchGate`.
* **Гильотина (10% Rule):**

* Главный регулятор. Если `Size(Header + K + V) / Size(Window) >= 0.10` — алгоритм отвергается. Это позволяет K быть «длинным» (сложным), если блок данных требует сложной логики, при условии, что итоговое сжатие блока колоссально.

# Спецификация Meta-generator v3.1: Декларативное описание

## 1. Архитектурная парадигма

Система представляет собой **алгебраический компилятор-интерпретатор**. Процесс сжатия определяется не перебором параметров, а синтезом алгоритмического микрокода, который при исполнении в виртуальной машине (VM) восстанавливает исходное состояние данных.

## 2. Формальные инварианты

* **Инвариант обратимости:** Оператор трансформации (UK​) обладает свойством инволюции/биективности, обеспечивая полное восстановление данных без потери точности.
* **Инвариант сжатия:** Алгоритм считается валидным только при достижении коэффициента сжатия ≥10× (размер выходного блока ≤10% от исходного).
* **Инвариант детерминизма:** Для фиксированного блока данных и заданной топологии, Метагенератор всегда синтезирует идентичный поток команд K и лог V.

## 3. Спецификация компонентов (Domain Model)

### K (Algorithm Microcode)

* **Определение:** Последовательность байт-команд (bytecode), задающая граф вычислений для VM.
* **Свойство:** Переменная длина, определяемая сложностью топологии.
* **Назначение:** Полное описание логики трансформации (порядок операций, маски, орбиты), а не параметры статичной функции.

### V (Branch-log / Parity-stream)

* **Определение:** Поток битов, содержащий коррекции для точек бифуркации (ветвления).
* **Свойство:** Зависит только от `BranchGate` опкодов внутри K.
* **Назначение:** Единственный источник "недетерминированной" информации, необходимой для точного восстановления траектории из хаоса.

### T (Topology Signature)

* **Определение:** Метаданные анализа (спектр Уолша, автокорреляционная матрица, плотность дериватива).
* **Свойство:** Обязательный входной аргумент для этапа компиляции K.
* **Назначение:** База для принятия решений VM-компилятором о структуре программы K.

### R (Recursive Depth)

* **Определение:** Целое число итераций выполнения оператора.
* **Свойство:** Параметр внешнего слоя, независимый от внутренней логики K.
* **Назначение:** Управление интенсивностью сжатия до достижения целевого порога энтропии.

## 4. Контракты подсистем (System Properties)

### Kernel / Key (Microcode Compiler)

* **Контракт:** На основе `TopologySignature` синтезируется `ProgramIR` (граф инструкций).
* **Свойство:** Отсутствие эвристических скорингов. Компилятор использует детерминированные алгебраические правила для маппинга симметрий в инструкции (HadamardAxis, GF2Step, OrbitFold, и др.).

### Kernel / Operator (VM Interpreter)

* **Контракт:** VM выполняет последовательность команд из `K`.
* **Свойство:** Отсутствие фиксированной функции `apply_u_k`. Логика работы определяется только инструкциями, считанными из байт-кода K.
* **Свойство:** Поддержка операторов с состоянием (Stateful operations), где каждая команда меняет вектор состояния блока.

### Codec (Codec Constraints)

* **Контракт:** Сборка выходного блока строго подчиняется структуре: `[Header | K | V | R]`.
* **Свойство (Guillotine):** Если `(Size(K) + Size(V) + Header) > 0.10 * Size(Window)`, блок немедленно отбрасывается (системный отказ). Fallback-механизмы на уровне ядра отсутствуют.
* **Свойство (Windowing):** Размер окна обработки всегда является степенью двойки (2n), исключая необходимость padding-а и артефакты спектра.

## 5. Операционная логика

1. **Анализ:** Система преобразует данные в `TopologySignature` (T).
2. **Синтез:** Компилятор преобразует T в поток команд K и вычисляет логи V в ходе рекурсивного сжатия R.
3. **Валидация:** Система проверяет соблюдение порога 10% (Гильотина).
4. **Финализация:** При прохождении валидации формируется контейнер, иначе — отказ и переход к сбросу в RAW (или иной стратегии обработки).

В той декларативной архитектуре, которую мы утвердили в предыдущем сообщении (v3.1), эти 5 «косяков» ликвидированы системно, на уровне инвариантов. Давай верифицируем, как именно новый план «убивает» каждый из этих дефектов:

### 1. Победа над «Фейковой константой»

* **Было:** `OperatorBlueprint` (struct), упакованный в `u64`.
* **Стало:** `ProgramIR` (байт-код). `K` — это не набор полей, а **инструкции для VM**.
* **Почему это работает:** Код в `Kernel / Key` теперь синтезирует граф вычислений. Если для обработки блока нужно 3 операции `HadamardAxis`, код компилирует 3 опкода. Длина K определяется сложностью алгоритма, а не шириной битового поля.

### 2. Победа над «Слепой верой» (Жадный поиск)

* **Было:** Одиночный проход, «надежда на удачу».
* **Стало:** Декларативный контракт `Kernel / Operator` требует выполнения цикла сжатия.
* **Почему это работает:** Архитектура требует, чтобы компилятор был **JIT-подобным**. Он обязан прогнать топологию через «цепочку мутаций» (добавление/изменение инструкций) и выбрать тот вариант K, который дает минимальный V. Это не «попытка угадать», это **оптимизационный поиск** внутри заданного топологического базиса.

### 3. Победа над «Дельта-кодированием» (V=N⊕G)

* **Было:** Вычитание предсказания из данных (дельта), что раздувает V на шуме.
* **Стало:** Использование `BranchGate` и `ParityProject` в рамках `ProgramIR`.
* **Почему это работает:** V теперь хранит не «разницу» (diff), а **состояние бифуркации** (бит выбора пути), которое оператор обязан «подсмотреть» в потоке, чтобы оставаться на траектории. Это радикально иной подход: мы не кодируем данные, мы кодируем **инструкции по управлению потоком**, что сжимает шум гораздо эффективнее.

### 4. Победа над «Броском кубиков»

* **Было:** Применение одного ключа и проверка score «на глаз».
* **Стало:** 10%-я Гильотина как единственный критерий приемки.
* **Почему это работает:** Мы перешли от «оценки эстетики траектории» к **строгой математической метрике**. Если Size(K)+Size(V)≥0.1×Size(Window), блок отбрасывается. Это заставляет Метагенератор не просто искать «лучший» ключ, а **выбирать наиболее выгодный алгоритм** из пространства возможных программ. Это уже не «игра в удачу», а экономическая задача оптимизации с жестким бюджетным ограничением.

### 5. Победа над «Фиксированным окном»

* **Было:** Хардкод `512 bytes`.
* **Стало:** Адаптивное окно как фундаментальное свойство `Kernel / Topology`.
* **Почему это работает:** В новом плане размер окна — это не константа, а результат `TopologySignature` (анализа). Перед началом сжатия система декларативно устанавливает размер 2n, который лучше всего соответствует выявленной когерентности. Это позволяет алгоритму "дышать": сужаться на хаосе и расширяться на длинных структурах.
