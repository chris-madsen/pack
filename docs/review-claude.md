## 1. `OperationCode` — перечисление-пустышка

`OperationCode` в `base.rs` объявлен с 17 значениями и кодами (`0x01`–`0x11`), но **нигде не участвует в реальном исполнении**.

В `generate_base_from_root` из него строится `operation_schedule: Vec<OperationCode>` — но эта структура `GeneratedBase` затем нигде не используется в `codec.rs`. Кодек напрямую строит ключи через `build_operator_codec_key`, `build_trajectory_key`, `build_minimal_spectral_key` — **без всякого обращения к базе операций**.

Т.е. весь слой `base.rs` / `GeneratedBase` — это декорация. GPT написал красивую алгебру, но не подключил её ни к чему.

**Что должно быть:** `OperationCode` должен управлять тем, какие трансформации применяются к блоку перед кодированием. Каждый раунд обработки блока — это конкретная операция из расписания, которая меняет распределение данных так, чтобы затем алфавитное или спектральное сжатие дало выигрыш. Без этого никакой "генератор базы" не нужен.

## 2. Магическая константа — не та концепция

Ты правильно заметил. В коде `OperationCode` присваиваются коды чтобы... никуда не передаваться.

Магическая константа в смысле, о котором ты говоришь (как в `magic_example1.md`) — это **компактное описание всего битового алгоритма**, т.е. свёртка последовательности операций в одно число, которое однозначно воспроизводит преобразование. Правильный подход:

- каждая операция кодируется в фиксированное число бит (например, 5 бит = 32 возможных операции);
- параметры операции (сдвиг, маска, ротация) тоже упаковываются рядом;
- вся цепочка раундов сворачивается в одно число нужной разрядности.

GPT вместо этого сделал `u8` коды операций, которые нигде не используются как описание алгоритма. Сами же операции жёстко захардкожены в `build_operator_codec_key`:

```rust
let (xor_mask, rotate, phase_mask, shift_left, shift_right, odd_multiplier) =
    (1_u8, 1_u8, 0xF0F0_0F0F_AAAA_5555_u64, 13_u8, 7_u8, 0x9E37_79B9_7F4A_7C15_u64);
```

Это полная захардкоженность вместо вычисления.  Один и тот же оператор для всех блоков — это не метагенератор, это константа.

## 3. Метагенератор — фиктивный

`generate_operator_base_words` в `codec.rs` генерирует "базовые слова" для оператора через XorShift + FWHT-трансформацию.  Идея правильная: из ключа порождается псевдослучайная база, а реальные данные кодируются как "база + 1 крамб". Но:

**Проблема 1: Сид всегда берётся из первых 8 байт блока**

```rust
let first_word = block.get(..8)...;
let seed = u64::from_le_bytes(...) & !1_u64;
```

Это значит, что для блоков с одинаковым началом (логи, бинарные форматы с заголовком) ключ будет одинаковым — и base words совпадут. Никакой адаптации к топологии блока нет.

**Проблема 2: `apply_u_k` не является хорошим ГПСЧ для произвольных данных**
FWHT-оператор `apply_u_k` — инволюция (самообратна). Это хорошо для обратимости, но плохо для генерации базы: он не создаёт "плотного покрытия" пространства значений. GPT применяет его к последовательным индексам, а не к адаптированному спектру блока.

**Проблема 3: Условие совпадения base_word слишком жёсткое**

```rust
if (actual & !1) != *base {
    return Ok(None);
}
```

Оператор-кодирование срабатывает только если реальные данные совпадают с базой **с точностью до последнего бита**. Это чрезвычайно редкое условие для реальных данных. На практике оператор-блок никогда не активируется кроме синтетически созданных тестов `make_operator_shaped_block`.

**Что должно быть:** метагенератор должен искать такой ключ, при котором максимальное число слов блока попадает в поверхность `f(k, i) ± δ` с малым δ. Т.е. это оптимизационная задача — минимизировать суммарный объём остатка.

## 4. Спектральный предиктор — ГПСЧ, не спектральный

`synthesise_predictor_from_key` генерирует предиктор через XorShift-17:

```rust
state ^= state << 13;
state ^= state >> 7;
state = state.rotate_left(17);
*slot = (state as u8) ^ ((state >> 11) as u8) ^ ((state >> 37) as u8);
```

Это обычный линейный ГПСЧ. Он никак не связан со спектральным анализом блока. "Спектральным предиктором" он называется, но спектра не использует вообще. Настоящий спектральный предиктор должен восстанавливать блок через обратное FWHT от его наиболее значимых Walsh-коэффициентов — т.е. из `walsh_peaks` из `topology.rs`, которые реально вычисляются, но затем используются лишь для построения ключа, а не для генерации предиктора.

**Итог:** `compile_topology_to_key` строит ключ из пиков Уолша, но `synthesise_predictor_from_key` этот ключ читает и... игнорирует Walsh-конфигурацию, используя только `seed`. Связь разорвана.

***

## 5. Trajectory-блок — работает только на синтетических данных

`try_encode_trajectory_block` ищет блоки, где все 64-битные слова имеют **одинаковые старшие биты** (одинаковый `terminal`).

```rust
if !words.iter().all(|value| (*value >> steps) == first_terminal) {
    continue;
}
```

Это очень редкое условие. В реальных данных — логах, тексте, бинарных файлах — оно почти никогда не выполняется. GPT написал тесты `make_trajectory_shaped_block`, которые специально создают такие блоки — и они сжимаются в 10x. Но это не компрессор, это решатель частного случая.

**Нет адаптации:** не делается попытки найти аффинное преобразование, при котором условие выполнится для большего числа слов.

***

## 6. `build_minimal_spectral_key` — заглушка с нулями

```rust
KeySegment { kind: SegmentKind::PhaseMask, payload: 0_u64.to_le_bytes().to_vec() },
KeySegment { kind: SegmentKind::WalshConfig, payload: vec![0, 0] },
```

Нулевая фаза-маска и нулевые Walsh-сдвиги означают, что `apply_phase_reflection` ничего не делает (parity всегда 0), а оператор `apply_u_k` вырождается в тождественное преобразование. Предиктор будет просто XorShift-последовательностью, никак не связанной с блоком. Это fallback, написанный "лишь бы не падало".

***

## 7. `popcnt_profile` — используется формально, влияния ноль

В `analyze_topology` считается `popcnt_profile`, он попадает в `TopologySignature`, оттуда в `CrumbConfig` ключа.  Но в `synthesise_predictor_from_key` `CrumbConfig` не читается вообще. Вычисляется, сериализуется, игнорируется.

## 8. `fold_spectrum` / `fold_bits` — хэш без смысла

```rust
fn fold_spectrum(spectrum: &[f64]) -> u64 {
    spectrum.iter().enumerate().fold(0_u64, |acc, (index, value)| {
        let quantized = value.to_bits().rotate_left((index % 64) as u32);
        acc ^ quantized.wrapping_mul(0x9E37_79B9_7F4A_7C15)
    })
}
```

Это просто хэш-функция над спектром. Результат используется как `odd_multiplier` для оператора (`spectrum_fold | 1`). Проблема: хэш спектра — не то же самое, что информативный параметр оператора. Две совершенно разных функции могут дать одинаковый хэш, а значит один и тот же оператор — это отброс топологической информации.

Правильно: из доминирующих Walsh-пиков строить `odd_multiplier` как произведение (в смысле мультипликативной группы) коэффициентов, или брать из стандартного набора чисел, оптимальных по лавинному эффекту.

## 9. `spectral_seed_from_block` — берёт первые 8 байт, всегда

```rust
fn spectral_seed_from_block(block: &[u8]) -> u64 {
    let mut seed = 0_u64;
    for (index, byte) in block.iter().take(8).enumerate() {
        seed |= (*byte as u64) << (index * 8);
    }
```

Для блоков типа `\x00\x00\x00\x00...` seed будет 0, и тогда подставляется константа. Для JPEG/PNG заголовков seed будет одинаков для всех файлов одного формата. Настоящий адаптивный сид должен строиться из всего блока — например, как XOR свёртка или минимальный хэш.

***

## 10. Структура `FeistelKey` / `reversible.rs` — никуда не подключена

`feistel_forward` / `feistel_inverse` есть, тесты проходят, но в `codec.rs` ни одного вызова.  Весь Feistel-слой — мёртвый код. GPT написал его, протестировал, и не применил ни в одном из кодеков.
