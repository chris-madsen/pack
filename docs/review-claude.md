## Что реально исправлено (кредит там, где он есть)

`build_operator_codec_key` теперь вызывает `compile_topology_to_key(&analyze_topology(block)?)` — сид ключа зависит от всего блока, не только от его первых 8 байт.

## Что НЕ исправлено — фундаментальные дыры

### 1. Spectral-кодирование по-прежнему не кодирует

`encode_block` выбирает лучший из: Raw, Operator (trajectory после U_k), Trajectory, Alphabet.  Вариант `BlockEncoding::Spectral` **никогда не создаётся в encode-пути**. `SpectralBlock` существует только в десериализаторе и в одном тесте, где он собирается вручную. Функция `synthesise_predictor_from_key` вызывается только из `decode_spectral_block` — но encode-пути к Spectral нет совсем. Кодировщик физически не может выдать спектральный блок.

Это означает, что весь спектральный слой — `operator.rs`, `spectral.rs`, Walsh-анализ — существует как инфраструктура без пользователя. Компрессия идёт через Alphabet и Trajectory, которые в среднем дают посредственный результат.

### 2. `apply_operator_word_u_k_with_runtime` — это **инволюция**, не трансформация для сжатия

```rust
fn apply_operator_word_u_k_with_runtime(...) -> Result<u64, String> {
    let forward = feistel_forward(generated.apply_forward(value)?, feistel)?;
    // ...
    let reflected = (hadamard ^ phase).reverse_bits().rotate_right(rotate) ^ xor_mask;
    generated.apply_inverse(feistel_inverse(reflected, feistel)?)
}
```

Это подтверждается тестом `real_operator_word_u_k_is_an_involution`.  Функция применяется дважды и возвращает исходное значение — она симметрична. GPT использует её одинаково в encode и decode (`try_encode_operator_block` и `decode_operator_block` вызывают одну и ту же функцию).

Инволюция сама по себе не плохая идея — но смысл её применения к блоку перед trajectory-кодированием состоит в том, чтобы **улучшить структуру данных**: после трансформации слова должны иметь общий terminal и мало различающиеся нижние биты. Это обеспечивает сжатие. Но GPT **не проверяет, что U_k реально улучшила данные** — он просто применяет инволюцию и надеется на случайное выравнивание terminal.

Вероятность того, что случайный блок после U_k получит одинаковый terminal у всех слов при `steps=4` — примерно `1/16`. При `steps=2` — `1/4`. Для большинства реальных данных `try_encode_operator_block` вернёт `None`. Нет никакого поиска параметров, при которых трансформация **максимизирует число слов с общим terminal**.

### 3. Магическая константа — по-прежнему не то

В `compile_topology_to_key` `OperationCode`-ы кодируются в сегмент `Program` как последовательность байт.  Это **не магическая константа в смысле docs/magic_example1.md**. Настоящая магическая константа — это число, из которого детерминированно восстанавливается весь алгоритм преобразования. Здесь же:

- `Program` — просто `Vec<u8>` из кодов операций
- `WalshConfig` — отдельный сегмент с 10 байтами
- `PhaseMask` — ещё 8 байт
- `RevMix` — ещё 2 байта
- `AuxConst` — ещё 8 байт

Итого ключ занимает ~40–50 байт из 64 разрешённых.  Это **не сжатое представление алгоритма**, а просто структурированный набор параметров. Настоящая магическая константа должна упаковывать выбор операций, параметры сдвигов и маску в одно 64-битное число, из которого алгоритм разворачивается детерминированно. У GPT этого нет.

### 4. `generate_operator_base_words` генерирует псевдослучайные слова, а не предиктор блока

```rust
let x = generated.apply_forward(seed ^ idx.rotate_left(...))?;
let x = feistel_forward(x, &feistel)?;
// ...
word = word.wrapping_mul(operator.phase.odd_multiplier | 1);
word = generated.apply_forward(word)?;
words.push(feistel_forward(word, &feistel)?);
```

Это ГПСЧ с ключом, зависящим от топологии входного блока. Но это **не предиктор блока** — он не минимизирует XOR-расстояние между predictor и реальными данными. Правильный спектральный предиктор строится обратным FWHT от доминирующих Walsh-коэффициентов самого блока, а не из ГПСЧ, посеянного хешем блока. Остаток (residual) при таком подходе действительно был бы разреженным. При текущем подходе residual = весь блок, XOR-нутый со случайной последовательностью. Размер residual = размер блока = нулевое сжатие.

### 5. Trajectory-кодирование работает только на искусственных данных

`try_encode_trajectory_block` требует, чтобы **все** 64-битные слова блока имели одинаковый `value >> steps`.  Для реальных данных это срабатывает только при очень маленьком `steps` (2 бита, т.е. только 4 старших бита совпадают у всех слов) — что даёт смехотворную экономию. Тест `trajectory_mode_falls_back_to_raw_when_not_profitable` прямо подтверждает, что на случайных данных это возвращает None.  Никакой попытки найти трансформацию, которая выровняла бы terminal, нет.

### 6. `CompressionPolicy::MVP` требует 10% выигрыша — это отсекает большинство блоков

```rust
pub const MVP: Self = Self {
    max_key_bits: 512,
    max_key_fraction: 0.25,
    minimum_gain_fraction: 0.10,
};
```

10% минимальный выигрыш — очень жёсткий порог для блоков, где сжатие может быть небольшим, но постепенным. На практике для большинства реальных данных ни Operator, ни Trajectory не преодолеют этот барьер, и всё упадёт в Alphabet или Raw.

### 7. Alphabet-кодирование — единственное, что реально работает

И оно работает ровно так, как работал бы простой huffman по алфавиту уникальных байт. Это эффективно для текста с малым алфавитом (ASCII цифры, буквы), но бесполезно для двоичных данных с 256 уникальными байтами — там `bit_width >= 8` и функция сразу возвращает `None`.
