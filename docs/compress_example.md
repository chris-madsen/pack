место того чтобы мучить процессор делением, мы превращаем наше исходное число N в конвейер. За один шаг мы будем откусывать от него по 64 бита (размер стандартного регистра `uint64_t`).

### Прямой ход (Сжатие гигантского числа N)

Допустим, на вход подается массив данных (число N) размером 1024 бита (128 байт).

1. Мы берем младшие 64 бита числа N через маску `N & 0xFFFFFFFFFFFFFFFF`.
2. Этот кусок данных становится нашей «жирной» хлебной крошкой Vi​.
3. Сдвигаем число N вправо на 64 бита: `N = N >> 64`.
4. Повторяем, пока число не превратится в 1 (или 0).

**Результат вычислений:**

* Для файла в 1024 бита цикл выполнится всего **16 раз**!
* На выходе у нас конечное число (например, 1) и вектор из 16 крошек, каждая по 64 бита.

## Как это превратить в реальное сжатие? (Элементарный XOR-сдвиг)

Схема выше — это просто перекладывание бит из одного кармана в другой (сжатия нет, разность нулевая). Чтобы появилось сжатие, нам нужно сделать шаг детерминированным.

Давай добавим простейшую битовую операцию, например **XOR-сдвиг**, который используют в сверхбыстрых генераторах псевдослучайных чисел (Xorshift).

### Алгоритм «Распаковки» (Обратный ход)

Мы берем базовое число (единицу) и восстанавливаем терабайты данных, используя короткий вектор инструкций.

Пусть наша обратная функция делает следующее:

1. Берем текущее число `X`.
2. Делаем быстрый XOR-сдвиг: `X = X ^ (X << 13); X = X ^ (X >> 7);` (это генерирует лавинообразное расширение бит).
3. Накладываем «хлебную крошку» (битовую маску) через `OR` или `XOR`, чтобы скорректировать траекторию.

```
[Текущее число X] ──> [ Сдвиг << 13 ] ──> [ XOR ] ──> [ Сдвиг >> 7 ] ──> [ XOR ] ──> [ Наложение крошки V ] ──> [Новое число X (в 2 раза больше)]

```

За счет того, что `<< 13` удваивает и расширяет битовую сетку, число растет экспоненциально на каждом шаге.

### В чём профит по тактам CPU и битам?

Представь, что мы сделали всего **10 шагов** такого цикла.

* **Объем вычислений:** 10 итераций. Внутри каждой — пара сдвигов и пара XOR. Для процессора это буквально **30-40 тактов** (наносекунды).
* **Что произошло с данными:** Число за счет левых сдвигов выросло с 1 бита до, например, 10 000 бит.
* **Разность:** \* Мы сохранили вектор крошек размером всего в 100 бит (по 10 бит на шаг для коррекции хаоса).

* Получили число в 10 000 бит.
* Чистая разность по абсолютной величине: **9 900 бит**.

## Почему это идеальный компромисс для железа?

1. **Идеальное попадание в кэш процессора (L1):** Все операции происходят внутри регистров общего назначения (`RAX`, `RBX` и т.д.). Процессору даже не нужно ходить в оперативную память за промежуточными значениями.
2. **Векторизация (SIMD):** Если нужно распаковывать много таких последовательностей, современные процессоры через инструкции AVX-512 могут выполнять битовые сдвиги над восемью 64-битными числами *одновременно* за один систолический такт.
3. **Предсказуемость циклов:** Количество шагов фиксировано и известно заранее (например, строго 16 или 32 шага). Процессор идеально предсказывает переходы (Branch Prediction), конвейер не простаивает.

В итоге мы получаем «взрывной» рост объема данных из маленького зерна при практически нулевой нагрузке на процессор.
