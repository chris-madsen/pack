# План исправления `pack` после полного аудита

## 1. Итоги code review

### Критические дефекты

1. **Текущий `master` не компилируется.**
   - `src/domain/mod.rs` ошибочно заменён содержимым `kernel/mod.rs`.
   - После временного восстановления экспортов обнаруживаются ещё минимум 10 ошибок несовместимости `key.rs`, `topology.rs`, `codec.rs` и тестов.

2. **Текущий `U_K` фактически является простым `XOR` с константой.**
   - CNOT-бабочка линейна.
   - `xor_mask` и ротация сокращаются при обратном преобразовании.
   - Для любого `x` выполняется `U_K(x) = x XOR C_K`.
   - Такой оператор сохраняет все XOR-разности между словами и не может улучшить terminal grouping.
   - Тесты operator-path проходят преимущественно вакуозно: если operator никогда не выбран, проверка ничего не доказывает.

3. **`V` не является логом бифуркаций оператора.**
   - После `U_K` каждый `u64` просто раскладывается на:
     - младшие `steps` бит в breadcrumbs;
     - оставшиеся биты в terminal.
   - Это перенос исходных 64 бит между двумя контейнерами, а не производящая траектория.
   - Сам `U_K` не читает и не порождает `V`.
   - Требование из ТЗ о consumption points внутри оператора не реализовано.

4. **Analyzer и runtime остаются математически несвязными.**
   - Analyzer делает real-valued FWHT всего 4096-битного блока.
   - Runtime применяет 64-битную CNOT-сеть отдельно к каждому слову.
   - `strongest_binary_word_peaks` агрегирует независимые word-transform, а не анализирует единое пространство блока.
   - Совпадение номера “пика” не означает совпадение математической семантики.

5. **Формат `K` снова сломан.**
   - ТЗ требует три spectral-пика; текущий `key.rs` хранит только два.
   - Внутренний kind-tag отнимает два бита, хотя mode уже задаётся контейнером.
   - Operator index урезан до 6 бит, хотя FWHT блока 4096 бит требует 12-битную координату.
   - Поля молча обрезаются масками.
   - Пустой spectral program и нулевая амплитуда корректно не запрещены.
   - Некорректный `log2` из архива может привести к опасному сдвигу.
   - API `MagicKey` рассинхронизирован с вызывающим кодом.

6. **Метагенератор остаётся выбором из маленького набора эвристических кандидатов.**
   - Используются 12 комбинаций binary peaks.
   - Кандидаты оцениваются постфактум по terminal palette.
   - В `K` остаётся `fingerprint_bias`, полученный из avalanche-хеша.
   - Это не прямой компилятор `Topology → executable K`, требуемый ТЗ.

7. **`B_STD` — неиспользуемый фасад.**
   - Реальный codec больше не использует `GeneratedBase`.
   - `WalshHadamard` и `BitPermutation` в базе реализованы одинаково через `reverse_bits()`.
   - `B_FILE`, `B_LAYER` и `K_ROOT` отсутствуют.
   - `PatternId` не согласован с реальными `BlockMode`.

8. **First-layer 10× нигде не является контрактом.**
   - Сейчас принимается любой выигрыш больше нуля.
   - При отсутствии выигрыша возвращается raw recursive container.
   - Нет ошибки или метрики, показывающей нарушение целевого требования.

### File-specific проверка

Форматных эвристик по расширению, MIME, имени файла, изображению или тексту в domain-коде нет. Вход действительно поступает как `&[u8]`.

Но:

- `Alphabet` и `SparseAlphabet` используют распределение байтов;
- они универсальны, но являются обычным статистическим byte-кодированием;
- именно они маскировали неработающий operator-path на noise;
- они не могут считаться подтверждением работы целевого метагенератора.

Решение: оставить их fallback/baseline, но измерять generator-driven режим отдельно.

### Статус замечаний из `reviews/`

Уже были исправлены по смыслу, но должны быть восстановлены после текущего merge:

- truthful `inspect_archive`;
- единый serializer для `encoded_size_of`;
- topology cache;
- gas limit;

Всё ещё актуальны:

- отсутствие operator-driven `V`;
- бесполезность spectral residual на noise;
- отсутствие прямого topology compiler;

- отсутствие честных quality-контрактов.

Устаревшие пункты:

- `GenerativeBlueprint`, `KeySpaceOptimizer`, `BifurcationStream` из Gemini-review в текущем коде отсутствуют;
- размещение обычных Rust unit-тестов в `#[cfg(test)]` модулях само по себе корректно.

## 2. Архитектура format/base v2

### Восстановление целостности

- Восстановить правильные domain exports.
- Синхронизировать `key`, `topology`, `operator`, `codec` и тестовые API.
- Ввести `CodecError` вместо строковых ошибок.
- Разделить 2300-строчный `codec.rs` на:
  - block strategy core;
  - container/wire codec;
  - recursive layer policy;
  - metrics.
- Не поддерживать чтение экспериментального v1.

### Типизированные `K`

Mode хранится в `BlockRecord`, а не внутри `K`. В Rust используются отдельные типы:

```text
SpectralKey(u64)
OperatorKey(u64)
TrajectoryKey(u64)
```

Это одновременно:

- сохраняет все 64 бита payload;
- исключает интерпретацию operator-key как spectral;
- соответствует ТЗ.

`SpectralKey` восстанавливается в точной раскладке ТЗ:

```text
3 × index[13]
3 × amplitude[5]
3 × sign[1]
peak_count[2]
block_log2[4]
tie_bit[1]
```

`OperatorKey` для блока 4096 бит:

```text
3 × peak_index[12]
3 × peak_sign[1]
primary_shift_minus_1[5]
round_count_minus_1[3]
derivative_density[8]
popcnt_density[8]
phase_parity[1]
```

Все конструкторы выполняют checked validation; молчаливое усечение запрещается.

### Единое математическое пространство

- Рабочее состояние operator-path: единый `Block4096` из 64 `u64`.
- Topology extractor использует детерминированный integer FWHT над всеми 4096 битами.
- Никаких `f64` в канонической компиляции `K`.
- Walsh coordinate однозначно отображается в координату `(word, bit)` runtime.
- `OperatorKey` строится ровно один раз прямой функцией:
  `TopologySignature → OperatorKey`.
- Удалить candidate-family, fingerprint bias и последующий подбор ключа.

### Исполнимый `U_K`

Реализовать настоящий блоковый оператор:

```text
U_K = F_K⁻¹ · H_bin⁻¹ · D_K · H_bin · F_K
```

- `F_K` — нелинейная обратимая ARX/Feistel-композиция, напрямую развёрнутая из полей `OperatorKey`.
- `H_bin` — одна зафиксированная обратимая 4096-битная butterfly-сеть.
- `D_K` — phase mask, непосредственно построенная из трёх Walsh coordinates.
- `F_K` и `F_K⁻¹` имеют отдельные реализации.
- Запрещены `reverse_bits()` в качестве фиктивного Walsh/bit-permutation.
- Добавить контракт, запрещающий деградацию `U_K` в `x XOR constant`.

### Настоящий trajectory `V`

До реализации codec-path формализовать в ТЗ точный переход:

```text
contract_K(state) -> (next_state, optional_branch_bit)
expand_K(next_state, branch_bit) -> state
```

Обязательные свойства каждой trajectory rule:

- точки чтения `V` определяются только `K` и инвариантами раунда;
- encoder и decoder посещают одинаковые bifurcation points;
- `V` не является `N XOR G`;
- `V` не является механическим переносом отброшенных бит;
- `terminal_bits + V_bits` должен реально уменьшаться;
- для каждой rule приводятся прямой псевдокод, обратный псевдокод и доказательство однозначности.

Текущий `encode_parity_trajectory(value >> steps)` удалить из целевого operator-path. Его можно оставить только как диагностический baseline.

Если `docs/compress_example.md` не удаётся превратить в точную contraction rule с честным битовым балансом, он помечается как иллюстративный пример, а не доказательство работающего компрессора.

## 3. Экономика, блоки и режимы

- Первый слой принимается только при полном коэффициенте не хуже **10×**:
  `layer_1_size <= input_size / 10`.
- Каждый следующий слой принимается только при уменьшении полного контейнера минимум на **10%**.
- При меньшем выигрыше рекурсия останавливается и сохраняется предыдущий слой.
- Raw-блоки допустимы внутри слоя, но весь первый слой всё равно обязан пройти 10×-контракт.
- Если первый слой не достигает 10×, strict API возвращает `InsufficientFirstLayerGain`, а не выдаёт успешный raw-архив.
- Alphabet/SparseAlphabet остаются fallback для общего packer-path.
- Добавить отдельный `generator-only` профиль без alphabet, чтобы они не маскировали качество метагенератора.
- MVP по умолчанию использует 512-байтный блок.
- Убрать упаковку одного входа всеми возможными block sizes.
- Adaptive sizing реализовать как отдельный детерминированный `BlockSizer`, выбирающий размер по предсказанной стоимости `K + V + overhead`, без полного повторного encode каждого размера.

## 4. TDD и режим запуска тестов

Сначала добавить все красные контракты, затем исправить весь код. Полный suite не запускать после каждой небольшой правки.

### Быстрые проверки во время работы

- `cargo check --lib`
- точечные `cargo test module::test_name`
- `make test-fast` для чистых kernel/unit-тестов

### Новые обязательные контракты

- проект собирается после clean checkout;
- все три типа `K` имеют точную 64-битную раскладку;
- malformed `K` не вызывает panic;
- integer FWHT детерминирован;
- topology coordinates совпадают с runtime coordinates;
- `U_K(U_K(x)) = x`;
- `U_K` не является XOR-трансляцией;
- изменение каждого поля `OperatorKey` меняет соответствующую часть алгоритма;
- direct compiler не перебирает семейство ключей;
- operator сам инициирует чтение каждого бита `V`;
- изменение/удаление любого crumb либо меняет результат, либо делает поток невалидным;
- terminal и crumbs не являются механическим разбиением исходного слова;
- exact wire-size совпадает с бюджетом;
- decoder limits защищают длины, шаги, слои и allocation;
- deterministic input даёт побитово одинаковый архив;
- никакие имена, расширения или типы файлов не поступают в domain core.

### Организация тестов

- Оставить маленькие unit-тесты рядом с чистыми функциями.
- Перенести codec/container/e2e в `tests/`.
- Добавить property-based проверки ключей, оператора, parser и roundtrip.
- Удалить вакуозные тесты вида `if let Some(operator)`.
- Тест обязан явно требовать нужный mode либо явно проверять отказ.

### Единственные полные прогоны

1. После завершения всех исправлений из review.
2. После исправления найденных интеграционных ошибок.
3. Финальный release-прогон перед отчётом.

## 5. Финальная проверка и отчёт

Добавить рекурсивный benchmark runner, который обрабатывает все файлы в `tests/`, включая `tests/noise/`, не анализируя заранее их содержимое.

Для каждого файла выводить:

- original/packed bytes;
- общий factor;
- factor первого слоя;
- размеры последующих слоёв;
- выбранные block sizes;
- количество блоков каждого mode;
- отдельно `K_bits`, `V_bits`, terminal bits и overhead;
- долю generator-driven блоков;
- encode/decode time;
- roundtrip.

Финальные acceptance criteria:

- `cargo fmt --check`;
- `cargo check --all-targets`;
- полный test suite;
- byte-for-byte roundtrip каждого файла;
- первый слой каждого тестового файла не хуже 10×;
- каждый дополнительный слой даёт минимум 10%;
- итоговый packer не хуже 10× на каждом файле;
- отдельный generator-only прогон также достигает 10×, иначе метагенератор не считается реализованным;
- результаты README/TЗ обновляются только после фактического benchmark;
- создать review-resolution matrix для каждого пункта всех пяти файлов из `reviews/`.

Новые документы и изменения репозитория оформлять на английском согласно repository policy.
