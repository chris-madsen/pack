Если принять этот документ в текущем виде, мы вернемся к старым архитектурным тупикам. Давай разберем по пунктам, что здесь полностью сломано и что авторы плана упустили.

### 1. Полный провал с адаптивным окном (Пункт 7)

* **Что написано в плане:** *«evaluate only a small deterministic ladder such as: 4096, 8192, 16384 bits»*.
* **В чем косяк:** Это жесткий хардкод статической сетки, который полностью перечеркивает динамическую структуру. В нормальной архитектуре `v4.1` мы жестко зафиксировали: для **каждого конкретного прохода** (`Pass`) сначала аналитически определяются динамические границы `W_min(pass)` и `W_max(pass)`. А затем рабочий набор окон этого прохода строится **строго по удвоению между ними**. Предложенный в тексте «ladder» из трех фиксированных значений убивает всю адаптивность на корню.

### 2. Регрессия к фиксированным 64 битам для `K` (Пункт 1)

* **Что написано в плане:** *«Use a fixed-width program shape that still fits in 64 bits: 2 bits length, 4 × 4 bits opcodes, 4 × 10 bits operands...»*. В тестах опять требуют: *«K program packing/unpacking is exactly 64 bits»*.
* **В чем косяк:** Это откат к старой логике. Мы договорились, что константа `K` имеет **переменную длину (variable-length bit-packed stream)**. Её размер ограничен сверху только экономикой сжатия конкретного блока, а не типом данных или фиксированной сеткой. Если топология блока простая, `K` может и должна занимать всего 12 или 16 бит. Зашивать в нее жесткую структуру из 4 инструкций по 10 бит на операнд — это преступление против плотности упаковки.

### 3. Попытка протащить скрытый брутфорс (Пункт 2)

* **Что написано в плане:** Описывается «жадный экономический фильтр», который по очереди добавляет инструкции-кандидаты, а затем: *«after each append, run the current transform on the block and compute exact strict budget»*.
* **В чем косяк:** Авторы уверяют, что это "search-free", но на самом деле запускать полное выполнение трансформации над всем блоком данных (`run the current transform`) внутри генератора для проверки каждой гипотетической инструкции — это классический итеративный мутационный поиск (greedy search). Настоящий **прямой синтез (Direct Synthesis)** должен работать в один проход: FWHT выдал спектр -> спектральные пики напрямую смапились в биты параметров конвейера -> ключ собран. Без прогона данных туда-обратно ради оценки стоимости на каждом шаге.

### 4. Опасное возвращение к концепции Виртуальной Машины (Пункт 1 и 4)

* **Что написано в плане:** Постоянно используются термины *«instruction blueprint»*, *«opcodes»*, *«program length»*, а декодер заставляют *«interpret K as generate the transform graph»*.
* **В чем косяк:** Этот подход прямо ведет к созданию интерпретатора внутри декодера. Если в рантайме декодирования появится цикл, который читает «опкоды» и делает `match opcode { ... }`, branch predictor современного процессора сойдет с ума на высокоэнтропийных данных. Конвейер CPU будет постоянно сбрасываться, и скорость упадет.
* **Как должно быть:** Никаких VM и динамических графов инструкций в рантайме. Мясорубка должна быть **монолитным, прямолинейным branchless-кодом на Rust**. Биты из потока `K` — это не «команды для выполнения», это просто жесткие параметры (маски фаз, шаги циклических сдвигов, множители), которые подставляются в инвариантную битовую трубу. Данные должны лететь по одной и той же траектории, меняются только математические коэффициенты слоев.

2. **Пункт 1 (`K`):** Стереть упоминание фиксированных 64 бит и жесткой сетки операндов. Прописать: `K` — это `variable-length bit-packed stream`.

4. **Архитектура:** Добавить жесткое требование **Branchless Pipeline** — никаких VM, никаких `opcodes`, только монолитная битовая труба.
